## 1. この文書の役割

本書は、問題6、12、18の定性分析から得た抽象的特徴を「こくごのたね」へ取り込むための、依存関係付き実装計画である。

仕様の正本は次に残し、本書へ重複して確定仕様を書かない。

- 測定能力、測定対象外、本文構造と設問セットの境界：[item-blueprint.md](item-blueprint.md)
- 設問パターン、根拠グラフ、解答欄：[question-pattern-expansion.md](question-pattern-expansion.md)
- 本文・設問・正答根拠の生成手順：[algorithm-draft.md](algorithm-draft.md)
- 参照資料、権利状態、再利用可能範囲：[reference-anchor-registry.md](reference-anchor-registry.md)
- 実装済み範囲と検証状態：[implementation-progress.md](implementation-progress.md)

本書の役割は「何を、なぜ、その順番で実装するか」と、各段階の完了条件を示すことである。設計判断が変わった場合は、先に上記の正本候補を更新してから本書の順序を更新する。

## 2. 結論

新しい設問を個別に追加する前に、本文中の人物、出来事、感情、反応、参照対象、関係変化を結び付ける**物語状態・根拠グラフ**を実装する。

現行実装の根拠契約は、意味役割から1つの `SentenceId` を得る形が中心である。このままでは、次の問いを安定して生成・検査できない。

- 離れた場面の原因と結果を結ぶ問い
- 複数の原因欄を埋める問い
- 条件に合う出来事を漏れなく挙げる問い
- 指示語や抽象表現の参照先を説明する問い
- 空所の前後双方と整合する発話・心内語を選ぶ問い
- 人物の感情や人間関係が変化した理由を追う問い

したがって、実装順序は次の依存関係に従う。

```text
物語状態・根拠グラフ
  ↓
人物役割・感情・関係・伏線回収の本文構造
  ↓
新しい設問パターンと採点契約
  ↓
新しい解答欄
  ↓
設問セットへの組込み
  ↓
コーパス、ブラウザ、印刷、人間レビュー
```

## 3. 問題6、12、18から取り込む抽象仕様

参照元の人物、舞台、出来事、文章、選択肢、展開順は生成素材へ転用しない。取り込むのは、測定能力、証拠の種類、人物間の作用、回答操作だけである。

| アンカー | 観察した認知操作 | 取り込む仕様 | 現行との差 |
| --- | --- | --- | --- |
| 問題6 | 自己認識と周囲の評価のずれ、第三者の反応、公の場での感情変化を読む | 人物ごとの知識・評価、介入の意図と実際の効果、社会的圧力を表現する | `story-late-arrival.v1` は第三人物の参加時点を扱うが、介入の意味が主に協力へ固定されている |
| 問題12 | 序盤の期待が離れた終盤で回収され、感情と行動が変わる過程を追う | 複数段階の感情状態、離れた証拠、伏線と回収、複数原因、出来事集合 | `causal-trace-6q.v1` は6問あるが、複数空欄、全件列挙、文脈挿入、伏線回収は未実装 |
| 問題18 | 関係の悪化から修復までを、相手の状態の観察と行動選択から読む | 人間関係の状態遷移、抽象表現の参照先、修復行動、様態語の文脈適合 | 現行本文構造は課題解決が中心で、関係自体を状態として保持しない |

## 4. 目標とするドメイン契約

### 4.1 二段階の根拠グラフ

blueprintが本文を作る段階では、まだ確定した `SentenceId` がない。そのため、次の二段階に分ける。

1. `EvidenceGraphDraft`：blueprint固有の文の役割を参照する。
2. `EvidenceGraph`：本文組立て後に役割を `SentenceId` へ解決する。

型名と配置は実装時に確定するが、少なくとも次の情報を持たせる。

```ts
type EvidenceNodeKind =
  | "event"
  | "emotion"
  | "reaction"
  | "utterance"
  | "referent-target"
  | "relationship-state";

interface EvidenceNodeDraft {
  readonly id: string;
  readonly kind: EvidenceNodeKind;
  readonly sentence_roles: readonly string[];
  readonly character_ids: readonly string[];
  readonly phase_id: string;
  readonly explicitness: "explicit" | "inferred";
}

interface EvidenceEdge {
  readonly from: string;
  readonly to: string;
  readonly type:
    | "precedes"
    | "causes"
    | "refers-to"
    | "changes-emotion"
    | "changes-relationship"
    | "resolves";
}
```

さらに、単一ノードでは表せない問いのため、次を別契約として持つ。

- `EvidenceCollection`：全件列挙の対象、順序、部分点、完全性条件
- `StoryCallback`：序盤の設定ノード、終盤の回収ノード、両者の関係
- `ParticipantState`：場面ごとの参加人物、知識、意図、他者への評価
- `InterventionEffect`：介入者の意図と、主人公・状況へ生じた実際の効果

### 4.2 既存パターンとの互換性

根拠グラフ導入時に、現行8パターンの本文・正答・表示を同時に作り直さない。

- 現行の意味役割から `SentenceId` を引く処理は互換アダプターとして残す。
- 新しい設問だけが複数ノード、辺、集合、伏線回収を要求する。
- 現行seedで、題名、本文、設問順、正答が意図せず変わっていないことを回帰検査する。
- 生成来歴へグラフ版を追加し、将来の互換性判断に利用する。

## 5. 追加する本文構造

### 5.1 `story-social-feedback.v1`

問題6から、人物数そのものではなく、自己認識と他者評価のずれを抽象化する。

必須契約：

- 主人公が自分または自分の行動をどう捉えているかを持つ。
- 別人物が同じ場面をどう捉えるかを持つ。
- 他者の反応が主人公の感情または次の行動を変える。
- 介入の意図と実際の効果を別々に記録できる。
- 同じ人物が常に助け役・妨害役にならない。

### 5.2 `story-delayed-callback.v1`

問題12から、序盤と終盤を離して読む必要のある構造を抽象化する。

必須契約：

- 序盤に期待、約束、助言、未完了の情報のいずれかを置く。
- 中間場面だけでも局所的な出来事が成立する。
- 終盤の合図、再登場、想起、発見のいずれかが序盤ノードを回収する。
- 回収によって主人公の感情または行動が変わる。
- 設定と回収が同じ文・隣接文へ縮退しない。

### 5.3 `story-relationship-repair.v1`

問題18から、人間関係の状態変化を抽象化する。

必須契約：

- 初期関係、関係を変える出来事、変化後の関係を別ノードにする。
- 主人公が相手の行動・表情・発話のいずれかを観察する。
- 観察が主人公の認識または行動選択を変える。
- 修復は片方の心情だけで成立させず、相手の応答または関係状態の変化を残す。
- 特定の家庭構成、学校行事、遊び、約束違反を必須題材にしない。

## 6. 追加する設問と解答欄

| 実装順 | 設問パターン | 必要なグラフ情報 | 解答欄 | 採用理由 |
| ---: | --- | --- | --- | --- |
| 1 | `QP_REFERENT_EXPLANATION` | `referent-target` と `refers-to`、必要要素 | `AL_SHORT_OPEN` | 問題18で中心的な、抽象表現を具体化する操作が現行にない |
| 2 | `QP_CONTEXT_INSERTION_CHOICE` | 空所前後の状態、人物、感情変化 | `AL_CONTEXT_INSERTION` | 前後双方を読む必要があり、現行の次行動選択とは異なる |
| 3 | `QP_CAUSE_FRAME_MULTI_EXTRACT` | 複数の原因ノードと因果辺、意味役割順 | `AL_MULTI_SLOT_SENTENCE` | 問題12の因果を分解して読む操作と、異なる記入動作を導入できる |
| 4 | `QP_ENUMERATE_ALL_EVENTS` | `EvidenceCollection`、順序、完全性 | `AL_ENUMERATED_EXTRACTS` | 単一抜き出しでは測れない、条件に合う集合全体の探索を扱う |
| 5 | `QP_MANNER_CLOZE_CHOICE` | 動作、意図、速度・視線・秘密性等の意味特徴 | `AL_CHOICE_LIST` | 問題18の語の働きを扱う。ただし語彙候補の自然さを保証してから導入する |

`QP_EVENT_SEQUENCE_CHOICE` を `QP_CONTEXT_INSERTION_CHOICE` の実装済み扱いにはしない。また、`QP_CAUSE_RESULT_CHOICE` を複数空欄の因果抽出の代用にはしない。

## 7. 誤答とセット全体の契約

### 7.1 誤答の分類

新しい選択式問題は、不自然な語や明らかな冗談を誤答にしない。各誤答へ次の分類を記録する。

- 人物違い：別人物の心情・知識なら成立する。
- 場面違い：別の段階なら成立する。
- 部分原因：因果の一部分だけを捉えている。
- 根拠不足：もっともらしいが、本文に必要な証拠がない。
- 状態反転：変化前と変化後を取り違えている。

正答だけが長い、具体的、丁寧であることを禁止し、選択肢の表面特徴による正解推測を検査する。

### 7.2 設問セットの検査

- 少なくとも3つの物語段階から根拠を使う。
- 同じ一文だけで3問以上が解けない。
- ある設問の本文や選択肢が、別の設問の正答を直接教えない。
- 人物違い・場面違いの根拠IDを取り違えていない。
- 伏線回収を測るセットでは、設定側と回収側の両方を根拠に使う設問を1問以上含む。
- 選択、抜き出し、記述の筆記量を学年と紙面に応じて分散する。
- 問数を増やす場合も、設問数そのものを難度の根拠にしない。

## 8. 難度として記録する軸

次は実測難易度ではなく、生成物を比較するための構造値として記録する。

| 軸 | 記録例 |
| --- | --- |
| 感情遷移数 | 0、1、2、3以上 |
| 視点人物数 | 1人、2人、3人以上 |
| 同一場面の能動人物数 | 1人、2人、3人以上 |
| 根拠間距離 | 同一文、同一段落、複数段落 |
| 伏線回収距離 | 中間ノード数、段落数 |
| 関係状態の遷移数 | 安定、悪化、修復等の遷移数 |
| 根拠種別数 | 出来事、発話、反応、感情、関係等 |
| 誤答の近さ | 明白な矛盾、別場面、部分的整合 |

児童試行前に「小学1年生標準」「難度3」などの実測ラベルへ変換しない。

完了条件：自動検査と印刷確認に加え、少なくとも開発者本人の解き直し記録が残り、未実証の難度を断定していない。

## 10. 検証コマンドと確認物

各段階で関連テストを先に実行し、段階完了時には少なくとも次を実行する。

```bash
npm test
npm run typecheck
npm run build
```

加えて、`kokugo-no-tane` の固定seedコーパス、実ブラウザ表示、印刷PDFの全ページ画像確認を行う。正確なコマンドは実装時点の `package.json` と [implementation-progress.md](implementation-progress.md) に従う。

確認記録には次を残す。

- 対象の本文構造ID、設問セットID、seed
- 自動検査の件数と結果
- 目視した画面条件、PDFページ数、カラー・白黒の別
- 解き直しで見つかった別解、指示不足、答え漏れ
- 未確認範囲と、児童利用を許可していないこと

## 11. 今回取り込まないもの

- 参照問題の人物、動物、学校行事、会話、固有の出来事、展開順
- 元の設問文、選択肢、正答表現、指定字数、紙面寸法
- 参照問題を根拠とする学年標準・難度値の断定
- AIによる正答、採点要素、根拠グラフの無検証な確定
- 新しい設問セットIDを増やすこと自体を目的とした実装

## 12. 着手時の判断

最初のコーディング対象は**段階1：根拠グラフの基盤**とする。新しい本文構造や設問を同じ変更へ含めず、現行出力の互換性とグラフの参照整合を確認してから、段階2以降を1つずつ縦切りで進める。

各段階は独立してコミットできる大きさに保ち、前段階の完了条件を満たしていない状態で後続の設問セットを公開レジストリへ追加しない。
