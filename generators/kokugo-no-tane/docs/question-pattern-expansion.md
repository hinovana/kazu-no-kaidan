## 1. 結論

解答用紙だけをランダムに変えても、出題のワンパターン感は解消しない。

先に「何を読ませ、どの根拠から、どの形式で答えさせるか」を表す**設問パターン**を選び、その結果から解答欄を決める必要がある。

```text
本文構造
  ↓
正答根拠グラフ
  ↓
設問セット設計を選ぶ
  ↓
利用可能な設問パターンを選ぶ
  ↓
正答・許容解・採点要素を先に確定する
  ↓
設問文・選択肢を作る
  ↓
回答形式に対応する解答欄を選ぶ
  ↓
設問固有検査と紙面検査を行う
```

問題6から抽出した標準4問は基準セットとして残した。その上で、現行の3つの本文構造が正答根拠を安全に供給できる範囲から、因果・文脈・結果・行動根拠型心情のパターンを追加した。現在は次の3種類をseedから決定的に選ぶ。

1. 問題6型：明示情報と心情推論を分ける標準4問
2. 問題12型の一部：因果と出来事をたどる6問
3. 問題12・18型の一部：文脈と行動根拠型心情を組み合わせる4問

指示内容、様子語、複数空欄、全件列挙は、現行本文へ後付けすると正答一意性を保証できないため未実装のまま残す。

## 2. 参照問題から抽出した違い

参照資料の人物、舞台、出来事、表現、選択肢、正答は生成素材にしない。ここで扱うのは設問形式と根拠契約の抽象的特徴だけである。

### 2.1 問題6

問題6は、現行の標準4問を設計した基準アンカーである。

| 観察した設問 | 抽象化した能力・操作 | 解答用紙上の違い |
| --- | --- | --- |
| 人物を表す語を指定字数で抜き出す | 条件に合う明示語を一意に特定する | 指定字数と同数のマス |
| 発話を受けた人物の気持ちを選ぶ | 場面と本文中の心内語を対応させる | 枠で囲まない3択 |
| 人物や集団の好みを抜き出す | 明示された事実を本文から特定する | 字数を漏らさない連続欄 |
| 身体反応から気持ちを書く | 行動と直前の出来事から心情を推論する | 短い無地の記述欄 |

この構成は、抜き出し、選択、抜き出し、記述という回答形式の基準になる。現行実装がワンパターンに見えるのは、本文構造が変わっても、この問題6型だけを繰り返しているためである。

### 2.2 問題12

問題12は、1つの物語に対して6種類の反応を求めている。

| 観察した設問 | 抽象化した能力・操作 | 解答用紙上の違い |
| --- | --- | --- |
| 原因を複数の空欄へ補う | 因果関係を分解し、複数の明示情報を対応する位置へ置く | 文の骨組みと複数の独立記入欄 |
| ある場面の気持ちを書く | 状況から心情を推論し、短く表現する | 字数を漏らさない無地の記述欄 |
| 参加した出来事をすべて抜き出す | 条件に合う本文情報を漏れなく列挙する | 正答件数に対応した複数記入欄 |
| 身体反応から気持ちを選ぶ | 行動・身体反応と心情を対応させる | 枠で囲まない選択肢と選択位置 |
| 本文の空所へ発話を入れる | 空所前後の状況と人物の発話を整合させる | 本文側の空所記号と選択欄 |
| 結果が生じた理由を選ぶ | 複数場面をつないで原因と結果を捉える | 因果関係を表す長めの選択肢 |

重要なのは、同じ「抜き出し」でも、単一語句を探す問題と「すべて」を列挙する問題では検査方法も解答欄も異なる点である。

### 2.3 問題18

問題18は4問だが、現行の標準4問とは構成が異なる。

| 観察した設問 | 抽象化した能力・操作 | 解答用紙上の違い |
| --- | --- | --- |
| 抽象的な表現の内容を説明する | 指示内容・言い換え対象を本文中から特定して説明する | 短い説明を書ける記述欄 |
| 日常的な感覚が変化した理由を選ぶ | 出来事と主観的な感じ方の因果を捉える | 因果を表す選択肢 |
| 空所へ様子を表す語を入れる | 動作、視線、心情に合う修飾語を選ぶ | 本文空所と語句選択欄 |
| 表情から気持ちを選ぶ | 行動と直前までの状況を組み合わせて心情を推論する | 行動根拠型の選択欄 |

問題18型は、明示された心情語の再認ではなく、指示内容、語の働き、行動からの推論を中心に組み立てられている。

## 3. 拡張前の実装がワンパターンに見えた理由

拡張前の実装には `story-retry-craft.v1` と `story-clue-discovery.v1` の2つの本文構造があった。しかし、どちらも問題6を基準にした次の標準4問へ固定されていた。

1. 人物を表す明示語の指定字数抜き出し
2. 明示心情の3択
3. 明示された行動の抜き出し
4. 状況と反応から心情を書く記述

本文の因果構造が変わっても、次がほぼ同じだった。

- 設問の順番
- 回答形式の並び
- 問われる能力の比率
- 解答欄の幅と形
- 問1から問4までの視覚的リズム

したがって、問題用紙では物語が変化していても、解答用紙を見ると同じ問題の差し替えに見えた。現在は、設問セットを本文構造とは別にseedから選び、4問または6問、設問順、解答欄の並び、先頭設問の根拠位置を変えている。

## 4. 設計を3層に分ける

### 4.1 設問セット設計

`question_set_blueprint_id` は、1枚の教材でどの能力と回答形式をどの順番で組み合わせるかを定める。

例：

```text
standard-reading-4q.v1
causal-trace-6q.v1
context-and-inference-4q.v1
```

### 4.2 設問パターン

`question_pattern_id` は、1問について次を結び付ける。

```text
測定能力
必要な本文根拠
正答の作り方
許容解
誤答の作り方
採点要素
設問固有の機械検査
既定の解答欄
```

### 4.3 解答欄レイアウト

`answer_layout_id` は印刷上の記入形式だけを扱う。

同じ選択式でも、短い心情語と長い因果説明では必要な列幅が異なる。同じ記述式でも、単一語句、複数列挙、短い説明、理由付き記述では同じ欄を使わない。

設問パターンは既定のレイアウトを持つが、紙面幅と学年に応じたレイアウト変形を許可する。正答字数などの非公開情報を解答欄から漏らしてはならない。

### 5.2 複数空欄の因果文

先に因果グラフを作り、そのノードを文の骨組みへ対応させる。

```text
場面・時点 → きっかけ → 外的出来事 → 主人公の受け止め
```

生成時に確定するもの：

- 空欄数
- 各空欄の意味役割
- 各空欄の正答
- 本文中の根拠ID
- 語形変化を許すか
- 空欄間の順序
- 部分点

機械検査：

- 各正答が本文に存在する。
- 同じ意味役割を満たす別候補が本文にない。
- 正答を別の空欄へ入れ替えても成立しない。
- 完成した因果文が本文の出来事と矛盾しない。
- 空欄の幅から正答字数を不必要に漏らさない。

### 5.3 すべてを挙げる問題

「すべて」と問う場合、本文生成時に対象集合を確定する。

```json
{
  "collection_id": "events_joined_by_protagonist",
  "member_evidence_ids": ["sentence_4", "sentence_7", "sentence_9"],
  "answers": ["...", "...", "..."],
  "order_policy": "passage_order",
  "partial_credit": "one_point_per_member"
}
```

本文に、条件へ部分的に合う紛らわしい出来事を無制御に追加してはならない。補助文を挿入したあとも集合が変わっていないことを検査する。

解答欄は正答件数に応じた独立欄を使う。ただし、件数自体を問う設計では欄数が答えを漏らすため、1つの連続欄へ切り替える。

### 5.4 文脈挿入問題

本文生成前に空所位置を予約し、空所前後の状態を構造化する。

```text
空所前: 出来事、人物の知識、直前の心情
空所: 発話または心内語
空所後: 行動、心情変化、結果
```

正答は前後の両方と整合しなければならない。誤答は次のどれかに対応させる。

- 空所前とは合うが空所後と矛盾する。
- 別の人物の心情なら成立する。
- 物語の別場面なら成立する。

正答だけが長い、丁寧、具体的など、表面特徴で当てられないようにする。

### 5.5 指示内容の説明

本文に抽象的なまとめ表現を置き、その前方に一意な具体場面を置く。

正答は単語ではなく、必要な出来事要素の組として保持する。

```text
誰が
何をする予定・約束だったか
実際に何が起きたか
それが関係へどう影響したか
```

設問ごとに必要要素を1〜3個へ限定する。模範解答の一文だけを唯一解とせず、要素充足で採点する。

### 5.6 様子を表す語の空所補充

候補語は日本語として不自然な語を混ぜるのではなく、次の誤り方を持たせる。

- 動作には合うが人物の意図と合わない。
- 心情には合うが対象となる動作を修飾できない。
- 別場面なら自然だが空所前後とは合わない。

本文構造モジュールは、空所対象の動作、視線、速度、秘密性などの意味特徴を宣言する。

## 7. 本文側に必要な根拠グラフ

新パターンは、完成した本文へ後付けしない。本文構造モジュールが、設問に利用可能な根拠、人物状態、関係状態、伏線回収を宣言する。根拠グラフ契約は採用済みだが、現在のコードは未実装である。

### 7.1 二段階の解決

本文構造モジュールが作る時点では確定した `SentenceId` がないため、次の二段階に分ける。

| 段階 | 型 | 文への参照 | 作成責務 |
| --- | --- | --- | --- |
| 本文構造内 | `EvidenceGraphDraft` | `sentence_roles` | blueprintがscenario、trait、本文役割列と一緒に作る |
| 本文組立て後 | `EvidenceGraph` | `sentence_ids` | 生成エンジンが文役割を確定IDへ解決して作る |

`EvidenceGraphDraft` のノードは、文役割の解決件数を `sentence_cardinality` として `one` または `one-or-more` で宣言する。単一文を要求するノードが0件または複数件へ解決された場合は生成を拒否する。`EvidenceGraph` へ解決した後は、設問が意味ノードから実際の本文箇所まで追跡できなければならない。

### 7.2 ノードと辺

ノードの最小契約は次とする。

```ts
type EvidenceNodeKind =
  | "event"
  | "emotion"
  | "knowledge"
  | "intention"
  | "evaluation"
  | "reaction"
  | "utterance"
  | "reference-expression"
  | "referent-target"
  | "relationship-state";

interface EvidenceNodeDraft {
  readonly id: string;
  readonly kind: EvidenceNodeKind;
  readonly sentence_roles: readonly string[];
  readonly sentence_cardinality: "one" | "one-or-more";
  readonly character_ids: readonly string[];
  readonly phase_id: string;
  readonly explicitness: "explicit" | "inferred";
}
```

辺は次の6種類に限定する。

| 辺 | 意味 |
| --- | --- |
| `precedes` | `from` の出来事・状態が `to` より意味上先にある |
| `causes` | `from` が `to` の発生理由として本文で支えられる |
| `refers-to` | `reference-expression` ノードが具体的な `referent-target` または出来事ノードを指す |
| `changes-emotion` | 出来事・発話・反応が人物の感情状態を変える |
| `changes-relationship` | 相互作用が二人以上の関係状態を変える |
| `resolves` | `from` の序盤設定・未完了情報を、`to` の終盤ノードが回収する |

`precedes` は本文上の出現順と矛盾してはならず、循環を許可しない。`causes` は単なる時間順序の別名にせず、本文から因果を説明できる場合だけ使う。`relationship-state` ノードは異なる `character_id` を2件以上持つ。

### 7.3 人物状態、集合、伏線回収

ノードと辺だけでは表しにくい要件を、次の補助契約で保持する。

| 契約 | 必須情報 | 用途 |
| --- | --- | --- |
| `StoryPhase` | `phase_id`、順序、その段階に属する文役割 | 人物状態の前後、伏線回収距離、場面違いの誤答を決定する |
| `ParticipantState` | `character_id`、`phase_id`、登場有無、知識・感情・意図・他者評価のノードID | 同じ人物の変化前後、人物違いの誤答を区別する |
| `InterventionEffect` | 介入者、`phase_id`、意図ノード、実際の効果ノード | 善意と結果、支援と社会的圧力などを分離する |
| `EvidenceCollection` | 集合ID、要素ノードID、本文順、完全性条件、部分点方針 | 「すべて」を問う設問で正答集合を固定する |
| `StoryCallback` | callback ID、設定ノードID、回収ノードID、間に必要な最小意味段階数 | 離れた場面の期待・約束・助言等と終盤の行動変化を結ぶ |

人物状態の各値は自由な説明文字列だけで保持せず、根拠ノードIDへ接続する。関係状態は `relationship-state` ノードとして保持し、初期状態、変化を起こす出来事、相手の応答、変化後の状態を `changes-relationship` で結ぶ。

`StoryCallback` は `resolves` 辺と対応し、設定と回収の間に一つ以上の意味段階を要求する。隣接する一段階因果を伏線回収として数えない。

### 7.4 下書き例

次はフィールド関係を示す最小例であり、現在の出力形式ではない。

```json
{
  "version": "evidence-graph.v1",
  "phases": [
    { "id": "setup", "order": 1, "sentence_roles": ["expectation"] },
    { "id": "middle", "order": 2, "sentence_roles": ["problem"] },
    { "id": "payoff", "order": 3, "sentence_roles": ["fact"] }
  ],
  "nodes": [
    {
      "id": "setup_expectation",
      "kind": "emotion",
      "sentence_roles": ["expectation"],
      "sentence_cardinality": "one",
      "character_ids": ["protagonist"],
      "phase_id": "setup",
      "explicitness": "explicit"
    },
    {
      "id": "later_action",
      "kind": "event",
      "sentence_roles": ["fact"],
      "sentence_cardinality": "one",
      "character_ids": ["protagonist"],
      "phase_id": "payoff",
      "explicitness": "explicit"
    }
  ],
  "edges": [
    { "id": "edge_setup_payoff", "from": "setup_expectation", "to": "later_action", "type": "resolves" }
  ],
  "callbacks": [
    {
      "id": "callback_1",
      "setup_node_id": "setup_expectation",
      "payoff_node_id": "later_action",

```

### 7.5 共通検査

- ノードID、辺ID、集合ID、callback IDがそれぞれ重複しない。
- `StoryPhase` のIDと順序が一意で、ノードが参照するphaseと本文役割が対応する。
- すべての文役割、ノード参照、人物参照、phase参照が存在する。
- `sentence_cardinality` と実際の解決件数が一致する。
- `precedes` と `resolves` が本文順に反しない。
- `precedes` に循環がなく、すべての辺で `from !== to` である。
- `causes`、`changes-emotion`、`changes-relationship` の接続先kindが契約と合う。
- `EvidenceCollection` の要素が重複せず、本文順と `order_policy` が一致する。
- `StoryCallback` の設定・回収ノードと対応する `resolves` 辺が存在し、最小意味段階数を満たす。
- 設問が要求する人物、場面、辺、集合、callbackをグラフが供給できない場合、その設問セットを選ばない。

本文構造ごとの段階1対応は次とする。

| 本文構造 | 段階1で下書き化する既存根拠 | 後続段階で追加する状態 |
| --- | --- | --- |
| `story-retry-craft.v1` | 期待、失敗、確認方法、反応、修正結果 | 人物状態、指示対象候補 |
| `story-clue-discovery.v1` | 注意、手がかり、確認方法、反応、発見理由 | 集合、観察時の人物状態 |
| `story-late-arrival.v1` | 問題、第三人物の初出、介入、主人公の決定、結果 | 介入の意図と実際の効果 |

段階1では現行根拠をグラフへ写し、本文構造や設問を変更しない。新しい人物状態、関係状態、伏線回収を必須にするのは、新しい `story_structure_id` を追加する後続段階とする。

## 8. 設問セット生成アルゴリズム

次は根拠グラフを実装した段階1以降の目標順であり、現在は手順4〜8が未実装である。

```text
1. 学年、本文長、本文構造、seedを確定する
2. 本文構造モジュールからscenario、trait、本文役割列を生成する
3. 同じモジュールからEvidenceGraphDraftを生成する
4. 下書き内の文役割、ノード、辺、人物、phase、集合、callback参照を検査する
5. 本文役割列を文章化し、各文へSentenceIdを割り当てる
6. sentence_rolesをsentence_idsへ解決してEvidenceGraphを作る
7. 解決件数、本文順、循環、集合、伏線回収を共通検査する
8. EvidenceGraphが供給できる設問セット設計候補を列挙する
9. seedから対応済み設問セットを決定的に選ぶ
10. 各設問パターンへ重複しない根拠ノードを割り当てる
11. 正答、許容解、誤答分類、採点要素を先に作る
12. 補助文挿入後に根拠グラフ、正答集合、一意性を再検査する
13. 設問文と選択肢を生成する
14. 回答形式から解答欄を割り当てる
15. 合計筆記量と紙面幅を検査する
16. 設問固有検査、セット検査、印刷検査を行う
```

設問セットIDは実装都合の値なので、初期UIでは利用者へ選ばせない。アルゴリズム生成ではseedから決定的に選び、生成根拠画面へ記録する。

人間レビューで差が理解できるようになった後、必要なら次のような利用者向け表現を検討する。

- いろいろな問い
- できごとをたどる問い
- ことばと気もちを考える問い

内部IDをそのままフォームへ表示しない。

## 9. 解答欄レイアウト案

現行実装は `AL_FIXED_CHAR_BOXES`、`AL_SINGLE_EXTRACT`、`AL_CHOICE_LIST`、`AL_REASON_AND_EMOTION` の4種類である。その他は対応する設問パターンとともに未実装である。

| `answer_layout_id` | 対応する回答 | 紙面上の原則 |
| --- | --- | --- |
| `AL_MULTI_SLOT_SENTENCE` | 複数空欄の因果文 | 文の骨組みと意味役割ごとの独立欄。字数マスは設問で字数指定した場合だけ |
| `AL_SHORT_OPEN` | 心情・指示内容の短い説明 | 1〜2本の無地縦書き欄。模範解答字数を漏らさない |
| `AL_ENUMERATED_EXTRACTS` | 複数項目の列挙 | 項目ごとに区切るが、件数を問う場合は連続欄へ切り替える |
| `AL_CHOICE_LIST` | 心情・因果・様子語の選択 | 丸数字と余白で選択位置を示し、選択肢を長方形で囲まない |
| `AL_CONTEXT_INSERTION` | 本文空所への挿入 | 空所記号を再掲し、選択肢は前後文を邪魔しない幅にする |
| `AL_REASON_AND_EMOTION` | 理由付き心情記述 | 理由と心情を分ける必要がある場合だけ領域を分ける |
| `AL_FIXED_CHAR_BOXES` | 指定字数の抜き出し | 設問で明示した字数と同数のマスだけを出す |
| `AL_SINGLE_EXTRACT` | 字数未指定の単一抜き出し | 正答字数を漏らさない連続した無地欄 |

### 9.1 紙面配分

現行の4列固定配置を、設問数と回答負荷から計算する。

```text
minimum_question_width
+ prompt_length_weight
+ expected_writing_weight
+ choice_length_weight
+ layout_specific_padding
= requested_width
```

全問を等幅にしない。6問セットでは、短い選択問題を細く、複数空欄と記述問題を広くする。1枚へ収まらない場合は文字を縮めず、設問ページを追加するか本文長・設問数を再選択する。

### 9.2 不要な補助表示を増やさない

次は現行方針を維持する。

- 「こたえ」「一つに○」「ぬき出し」などの欄種ラベルを付けない。
- 設問文そのものに回答方法を含める。
- 指定字数問題以外へ文字マスを出さない。
- 選択肢を個別の長方形で囲まない。
- 名前欄は解答用紙の右端に1か所だけ置く。

多様性は装飾や補助ラベルの追加ではなく、問いと記入動作の違いで作る。

## 10. 機械検査

### 10.1 根拠グラフ共通検査

段階1では、設問パターン固有検査より先に7.5の参照整合、解決件数、順序、循環、集合、伏線回収を検査する。失敗した場合は互換アダプターへ黙って戻さず、生成を失敗させる。グラフを利用する新設問では、設問が参照したノード、辺、集合、callbackから `SentenceId` まで到達できることも必須とする。

### 10.2 設問パターン固有検査

| パターン | 必須検査 |
| --- | --- |
| 複数空欄の因果文 | 各空欄の一意性、空欄間の入替不可、完成文と本文因果の一致 |
| すべてを挙げる | 正答集合との一致、本文補助文による集合変化なし、部分点合計 |
| 心情選択 | 行動・状況根拠の存在、正答1件、誤答分類、表面手掛かりの偏りなし |
| 文脈挿入 | 空所前後との両方向整合、正答1件、誤答が別場面へ対応 |
| 原因結果 | 原因ノードから結果ノードへの経路、一般常識だけで解けないこと |
| 指示内容 | 指示対象が一意、必要要素が本文に存在、許容解が要素採点可能 |
| 様子語空所 | 品詞一致、修飾対象一致、文脈上の正答1件 |

### 10.3 セット全体検査

- 同じ一文だけで3問以上が解けない。
- 同じ正答語句を複数問で使い回さない。
- 前の設問や選択肢が後の正答を教えない。
- 明示情報、因果統合、心情推論のうち少なくとも2系統を含む。
- 1年生の長い自由記述を連続させすぎない。
- 合計筆記量が本文長設定と独立して暴走しない。
- `question_set_blueprint_id`、各 `question_pattern_id`、`answer_layout_id` を生成来歴へ残す。
- 人物状態の変化を問う設問は、同じ人物の異なるphaseを参照する。
- 関係状態の変化を問う設問は、初期状態、相互作用、相手の応答、変化後の状態を参照する。
- 伏線回収を問う設問は、設定・回収の両ノードと `StoryCallback` を参照する。

### 10.4 紙面検査

- 設問番号と設問本文が重ならない。
- ふりがなと罫線が重ならない。
- 記述欄がページ外へはみ出さない。
- 6問でも読む順序が右から左へ一意に分かる。
- 文字を縮小して無理に1ページへ詰めていない。
- 白黒印刷でも選択位置と記入欄が分かる。

## 11. 難度パラメータ

回答形式の違いを、そのまま難度差と呼ばない。次を別々に記録する。

| 軸 | 例 |
| --- | --- |
| 根拠数 | 1か所、2か所、集合全体 |
| 根拠距離 | 同一文、同一段落、複数段落 |
| 推論段階 | 明示情報、1段階因果、2段階因果 |
| 回答生成量 | 選択、単語、短句、1〜2文 |
| 空欄数 | 1、2、3 |
| 列挙数 | 2、3、4 |
| 誤答の近さ | 明白な矛盾、別場面、部分的整合 |
| 人物状態遷移数 | 0、1、2、3以上 |
| 関係状態遷移数 | 0、1、2以上 |
| 視点人物数 | 1人、2人、3人以上 |
| 伏線回収距離 | 間にある意味段階数、段落数 |

現行の生成条件プロファイル1〜5は、各実装済みセットに1問ずつ含まれる `QP_EMOTION_OPEN` の根拠距離だけを表す。新しい設問セット全体の難度へ流用せず、設問パターン別の生成パラメータとして再設計する。

## 12. AI生成との関係

AIは設問文や完成した解答用紙を直接生成しない。

AI設計図を使う場合も、ローカル側が次を担当する。

- 利用可能な根拠ノードの検査
- 対応できる設問パターンの選択
- 正答・許容解・採点要素の確定
- 選択肢の意味契約検査
- 解答欄レイアウト
- 生成来歴

AI設計図に必要な根拠がない場合は、対応する設問パターンを出さない。空所や列挙対象を後からモデルに捏造させない。

## 15. 推奨方針

現行標準4問は削除せず、回帰基準として残した。設問パターンのレジストリ化を先に行い、標準4問を明示選択できる状態を守ったまま新セットを追加した。

利用者へ内部の設問セットIDを選ばせず、アルゴリズム生成ではseedから次の3つを決定的に選ぶ方針を推奨する。

```text
問題6型: 標準4問
問題12型: 因果をたどる6問
問題12・18型の一部: 文脈と気持ちを考える4問
```

3セットを大人が実際に解く人間レビューは継続する。次のコーディング対象は、指示内容・様子語・複数空欄・全件列挙を個別追加することではなく、現行3本文構造を二段階の根拠グラフへ写す段階1である。段階1では児童向け出力を変えず、参照整合と現行seedの回帰を先に確立する。
