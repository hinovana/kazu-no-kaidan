| 項目 | 値 |
| --- | --- |
| 文書状態 | 検討中 |
| 文書版 | `algorithm-spec.v0.9-draft` |
| 開発形態 | 個人開発 |
| 最終更新日 | 2026-07-19 |
| 実装状態 | 限定語彙・決定的テンプレート、本文構造2種類、設問セット3種類、解答欄契約、語彙候補DBの監査用投影、検証済みAI設計図アダプターによるブラウザプロトタイプを実装 |
| 実証状態 | 未実証・未校正 |

現行コードは `kokugo-no-tane.prototype.v0.9` である。seed付きの物語文、3種類の設問セット（4問または6問）、正答・根拠、設問パターン、解答欄、学年別漢字表記、範囲ごとの初出ふりがな、生成来歴、機械検査をブラウザ上で確認できる。本文長は「短め・ふつう・長め」から独立して選び、生成難度プロファイル1〜5は各セットの心情記述問題に使う根拠間距離を制御する。題材には動物を含み、テンプレートで指定した語句境界へ全角空白を入れる。画面は横書きのまま、印刷時はA4横・縦書き固定で、題名・本文・設問・選択肢・解答欄と解答用紙だけの名前欄を縦書きにする。1年生では既習判定にかかわらず漢字を含む語の範囲内初出へ必ずふりがなを付け、2・3年生では未習漢字を含む語の初出へ付ける。本文構造モジュールは、小さな失敗から修正する `story-retry-craft.v1` と、手がかりを確かめて発見へ至る `story-clue-discovery.v1` の2種類を登録し、アルゴリズム生成ではseedから決定的に選ぶ。各モジュールは題名、scenario変換、本文の役割列、正規化済み設問内容、正答根拠、構造固有検査を管理し、設問セット層が完成した設問と解答欄を構成する。ローカルAIプロキシから検証済み `story-plan.v1` を受け取った場合は、AIが供給した舞台・人物・目的・失敗・確認行動・解決を、対応する再挑戦型へ写像する。手がかり発見型のAI設計図契約は未実装である。語彙と読みは正式DBではなく、適用範囲を限定した `prototype_lexicon` を使うが、その11語は教育基本語彙候補DBの低学年区分と表記・読みで照合した小さな監査用投影を持つ。限定語彙に低学年一致がない語や高学年区分にしかない語を追加するとビルドを失敗させ、各生成物には候補DB版と照合結果を記録する。ただし候補語は全件人間未確認であり、この照合は生成許可や個別学年適合の保証ではない。テンプレート外の本文語句と文法の学年適合性も未検証である。機械検査に合格しても、人間未確認・未実証・未校正の `development_preview` であり、最初の人間レビューまでは児童に利用しない。

## 1. 文書概要

### 1.1 目的

本システムは、小学1〜3年生向けの国語読解練習問題を自動生成する。初版の文章種別は物語文に限定する。

生成対象は、単なる内容確認問題ではなく、次の能力を測定対象とする短文読解問題とする。

* 人物を表す明示語の探索
* 明示された心情の理解
* 発言や行動からの心情推論
* 本文中の情報の探索
* 根拠をもとにした短い記述

能力の操作的定義、能力間の境界、設問セットの構成は [item-blueprint.md](item-blueprint.md) と [question-pattern-expansion.md](question-pattern-expansion.md) を正本候補とする。本書の能力名だけを根拠に、実際にその能力を測定できると断定してはならない。

生成時には、参照問題の文章表現、登場人物、舞台、展開を模倣せず、**測定設計、根拠配置、設問形式、読み支援の抽象的特徴だけを再利用する**。参照元は [reference-anchor-registry.md](reference-anchor-registry.md) のアンカーIDで追跡する。

自動生成直後の問題は、機械検査に合格していても未校閲・未校正である。開発者本人による見直し手順と公開条件は [item-review-and-release.md](item-review-and-release.md)、児童試行と実測難易度は [calibration-and-fairness.md](calibration-and-fairness.md) に従う。

## 2. 対象読者

### 2.1 学年

小学1年生、2年生、3年生から選択可能とする。初期表示は小学1年生の夏相当とする。

学年は語彙・漢字・ふりがな・表記・生活経験の基準を決める。現行プロトタイプの生成難度プロファイル1〜5は、各設問セットの心情記述問題で使う根拠文どうしの距離を決める。本文長は「短め・ふつう・長め」から別に選択する。学年、生成難度、本文長は独立した入力とする。

生成難度は生成条件であり、児童にとっての実測難易度ではない。同じ数字を実測ラベルとして表示しない。

### 2.2 想定読解能力

* ひらがなをおおむね自力で読める
* 選択学年までに配当される漢字を一部またはおおむね読める
* 選択した本文長に応じて250〜750字程度の物語文を読める
* 本文中から語句や文を探せる
* 学年と生成難度に応じて1〜3文程度の記述ができる

### 2.3 読了時間

本文と設問を含め、標準的な児童が10〜15分程度で取り組める分量とする。

これは初期の設計目標であり、実測値ではない。学年・実施時期・表示条件ごとの所要時間を児童試行で確認する。

## 3. 生成物の構成

1回の生成につき、次の要素を出力する。

1. 題名
2. 本文
3. 設問4問または6問
4. 模範解答
5. 採点基準
6. 生成条件
7. 品質検査結果

児童向け出力では、4〜7を非表示にできるものとする。

## 4. 基本設計方針

### 4.1 本文先行型ではなく、読解構造先行型とする

自由に本文を書いたあとから設問を付ける方式は採用しない。

先に次の内部構造を生成する。

* 登場人物
* 性格
* 目的
* 期待
* 出来事
* 心情変化
* 行動
* 周囲の反応
* 設問対象箇所
* 正解根拠

この内部構造をもとに本文を生成する。

### 4.2 正解根拠を必ず保持する

各設問には、生成段階で次の情報を紐付ける。

```text
出題設計ID
設問セットID
設問ID
設問パターンID
解答欄ID
主たる測定対象
副次的に必要な能力
正答
根拠となる本文箇所
必要な推論段階数
許容される別解
誤答となる条件
```

正解根拠を特定できない設問は出力しない。

### 4.3 題材と読解スキルを分離する

題材は毎回変えてよいが、出題設計上の主たる測定対象は一定に保つ。

例：

```text
題材:
- 忘れ物
- 生き物の観察
- 友達との遊び
- 家の手伝い
- 工作
- 買い物
- 学校行事
- 動物

測定能力:
- 明示情報の探索
- 明示された心情の理解
- 心情推論
- 根拠を伴う短い記述
```

「動物の自慢話」など、参照問題の表層的特徴は固定しない。

### 4.4 学年・生成難度・本文長を分離する

```text
学年
→ 語彙・既習漢字・ふりがな・表記・題材経験

生成難度
→ 心情記述問題の根拠間距離（現行プロトタイプ）

本文長
→ 短め・ふつう・長めの文字数帯
```

`小学1年・生成難度5・短め` と `小学3年・生成難度1・長め` の両方を生成できなければならない。生成難度を上げるために上位学年の語彙を混入させてはならず、本文を長くするだけで推論段階を増やしたとみなしてはならない。

語彙・漢字・読みの作成根拠、データ構造、版管理、検証方法は [database-spec.md](database-spec.md) に従う。

## 6. ランダム生成仕様

## 6.1 ランダム化の対象

以下の要素を、それぞれ独立または条件付きで抽選する。

### 主人公

* 小学生
* きょうだい
* 家族
* 身近な動物
* 架空の生き物
* 擬人化された物

### 舞台

* 学校
* 家
* 公園
* 図書館
* 商店
* 電車やバス
* 海や川
* 畑
* 雨の日の町
* 架空の場所

### 主人公の性格

* 慎重
* 好奇心が強い
* 負けず嫌い
* 世話好き
* 恥ずかしがり
* あわてんぼう
* 自信家
* 我慢強い
* 思い込みが強い
* 素直

### 主人公の目的

* 成功したい
* 誰かを喜ばせたい
* 知りたい
* 褒められたい
* 困っている人を助けたい
* 自分の物を守りたい
* 仲間に入りたい
* 失敗を隠したい
* 約束を守りたい

### 中心事件

* 予想が外れる
* 失敗する
* 誤解される
* 忘れ物に気づく
* 誰かが困っている
* 大切な物が見つからない
* 思ったより難しい
* 知らなかった事実を知る
* 最初は嫌だったことのよさに気づく

### 心情変化

```text
期待 → 失敗 → 恥ずかしさ
不安 → 行動 → 安心
自信 → 予想外の出来事 → 反省
面倒 → 他者の反応 → 喜び
不満 → 理由の理解 → 納得
好奇心 → 発見 → 次の意欲
```

---

## 6.2 組み合わせ制約

完全な無作為抽選では、不自然な組み合わせが発生するため、制約付きランダム生成とする。

例：

* 「恥ずかしがり」な主人公に、冒頭から大勢の前で自慢させない
* 「誰かを助けたい」という目的に、理由なく物を隠す行動を組み合わせない
* 動物が主人公の場合、人間社会の制度を説明なく使用しない
* 対象学年の児童が単独では行いにくい行動を中心事件にしない
* 事件と心情変化の因果関係を明確にする
* 結末で性格が突然反転しない
* 教訓を説明文のように直接語らせすぎない

---

## 6.3 類似回避

過去の生成問題および参照問題との類似を避ける。

### 比較対象

* 主人公の種類
* 舞台
* 中心事件
* 心情変化
* 結末
* 特徴的な台詞
* 文の並び
* 設問の根拠位置

### 類似判定

次のうち3項目以上が一致する場合は再生成する。

```text
主人公カテゴリ
舞台カテゴリ
性格
中心事件
心情変化
結末類型
主要アイテム
対立相手の役割
```

文章表現の意味的類似度が一定値を超えた場合も再生成する。

### 禁止事項

参照問題から次の要素をまとめて流用してはならない。

* 動物園
* 自尊心の強い動物
* 観客に見せることを拒む展開
* 他の動物からからかわれる展開
* 失敗を見られて赤面する結末

個々の要素が偶然一致することは許容するが、物語骨格の一致は許容しない。

## 7. 本文生成仕様

## 7.1 文章構造

本文構造を生成エンジンへ直接埋め込まない。登録済み本文構造モジュールを `blueprint_id` で選び、モジュールが題名、scenario変換、本文の役割列、根拠配置、設問パターン用の `QuestionContent`、構造固有検査を提供する。完成した設問の順番、ID、解答欄は設問セット層が担当する。

生成エンジンは、seed、学年別表記、ふりがな、段落化、生成来歴、共通検査、ライフサイクル状態を担当する。本文構造モジュールは、これらの共通機能を再実装しない。

現行の `story-standard-4q.v1` は、本文構造ID `story-retry-craft.v1` として次の役割列を持つ。

| 順序 | role | 内容 |
| ---: | --- | --- |
| 1 | `opening` | 舞台、主人公、相手、作るものを提示する |
| 2 | `trait` | 主人公を表す明示語を置く |
| 3 | `expectation` | 作り始める目的または期待を置く |
| 4 | `explicit_emotion` | 明示心情問題の根拠を置く |
| 5 | `problem` | 主人公の小さな失敗を置く |
| 6 | `fact` | 修正のために決めた確認行動を置く |
| 7 | `inference_situation` | 失敗を相手に見られた状況を置く |
| 8 | `inference_reaction` | 心情記述問題の推論根拠となる反応を置く |
| 9 | `resolution` | 確認行動による修正結果を置く |
| 10 | `closing` | 二人の関係を損なわない終結を置く |

本文長と生成難度に応じた補助文は、モジュールが定める挿入可能位置にだけ追加する。補助文によって正答根拠の意味、順序、一意性を変えてはならない。

`story-clue-discovery-4q.v1` は、本文構造ID `story-clue-discovery.v1` として次の役割列を持つ。

| 順序 | role | 内容 |
| ---: | --- | --- |
| 1 | `opening` | 舞台、主人公、相手、観察対象を提示する |
| 2 | `trait` | 主人公を表す明示語を置く |
| 3 | `attention` | 主人公が小さな違いへ注意を向ける |
| 4 | `explicit_emotion` | 明示心情問題の根拠となる、調べる前の期待を置く |
| 5 | `clue` | 失敗ではなく、観察可能な手がかりを置く |
| 6 | `fact` | 手がかりを確かめる方法を主人公が決める |
| 7 | `inference_situation` | 複数の手がかりが一致する状況を置く |
| 8 | `inference_reaction` | 心情記述問題の推論根拠となる、顔を上げてうなずく反応を置く |
| 9 | `resolution` | 手がかりが生じた理由を説明する |
| 10 | `closing` | 二人の発見を損なわない終結を置く |

アルゴリズム生成では、本文構造IDと設問セットIDをフォーム項目にせず、正規化したseedのハッシュから本文構造2種類と設問セット3種類をそれぞれ決定的に選ぶ。これにより実装都合のIDを利用者へ選ばせず、同じ条件とseedの完全再現を保つ。明示IDは契約テストと開発時の比較にだけ使う。`story-plan.v1` は再挑戦型専用なので、AI生成時は手がかり発見型を抽選しないが、検証済み本文から設問セットは決定的に選ぶ。

---

## 7.2 文字数

本文長は生成難度から分離し、次の3段階から選ぶ。初期値は、従来より長い「ふつう」とする。

| 本文長 | 文字数帯 |
| --- | ---: |
| 短め | 250〜370字 |
| ふつう | 370〜570字 |
| 長め | 500〜750字 |

文字数は全角空白とふりがなの読みを除いた本文文字数で検査する。心情記述問題の根拠距離を満たすため、短めでも必要な補足文を優先する。帯は重なりを許し、同じseedでは `短め < ふつう < 長め` となることをテストする。学年または生成難度だけを変更して文字数帯を切り替えない。実測の読了時間から学年別補正が必要になった場合は、本文長設定とは別の補正値として版管理する。

---

## 7.3 一文の長さ

* 基本は10〜30文字
* 40文字を超える文は全体の10％以下
* 主語と述語の距離を長くしすぎない
* 一文に複数の出来事を詰め込みすぎない

---

## 7.4 段落

* 5〜12段落
* 会話文を適度に含める
* 1段落を長くしすぎない
* 心情変化の前後で段落を分ける

---

## 7.5 語彙

### 基本方針

* 選択学年の語彙データベースで使用可能な語彙を使う
* 難語は文章全体で0〜3語程度
* 難語を使う場合は、前後の文脈から意味を推測できるようにする
* 抽象語を連続させない

### 使用を抑える表現

* 大人向けの慣用句
* 二重否定
* 長い比喩
* 時系列が複雑な回想
* 指示対象が複数ある指示語
* 文脈なしでは理解できない専門語

---

## 7.6 漢字

次のいずれかの表記モードを選択可能とする。

### モードA：ひらがな中心

選択学年までの配当漢字も含め、ほぼひらがなで表記する。

### モードB：選択学年漢字使用

選択学年までに配当された漢字を使用し、それ以外はひらがなにする。

### モードC：ふりがな付き

選択学年より後に配当される漢字にはふりがなを付ける。

### モードD：初出ふりがな型

語の意味が選択学年の児童に理解できることを前提に、定義した範囲内の初回登場時にふりがなを付け、同じ語が再登場したときはふりがなを省く。1年生では既習・未習を問わず漢字を含む全語を対象にし、2・3年生では未習漢字を含む語を対象にする。

```text
初出：漢字と読みを対応づける
再出：初出で得た読みを思い出して読む
```

これは単なる表示上の補助ではなく、文章中の初出箇所を「読みの索引」として使う学習構造である。`ANCHOR-RUBY-Q12` と `ANCHOR-RUBY-Q18` で観察した構造を、初版の設計仮説として採用する。観察内容と限界は [reference-anchor-registry.md](reference-anchor-registry.md) に記録する。

標準設定はモードDとする。

### 初出ふりがな型の生成条件

語の難しさは、次の2軸に分けて判定する。

```text
語彙難度：耳で聞いたときに、選択学年の児童が意味を理解できるか
表記難度：現在の学期・教材進度で、その漢字表記を読めるか
```

次の順に表記を決める。

1. 語彙自体が選択学年に難しすぎる場合は、ふりがなの有無にかかわらず使用しない。
2. 1年生では、語彙が年齢相当で漢字を含むなら、既習判定にかかわらず範囲内の初出をふりがな付きで表記する。
3. 2・3年生では、語彙が年齢相当で漢字表記も既習なら、ふりがななしの漢字で表記する。
4. 2・3年生では、語彙が年齢相当で漢字表記が未習なら、範囲内の初出をふりがな付きで表記できる。
5. 読みが辞書にない固有名詞、特殊な読み、熟字訓は使用しない。

ふりがなは語単位で付ける。一文字ずつ機械的に分割せず、熟語・送り仮名・活用を含む正しい読みを語彙辞書から取得する。

### 初出判定の範囲

初出判定は、単純に問題セット全体で1回とはしない。

| 範囲 | 判定方法 |
| --- | --- |
| 題名 | 独立してふりがなを付け、本文の既出扱いにはしない |
| 本文 | 本文全体を1範囲とし、対象語の初出だけに付ける |
| 設問・選択肢 | 設問ごとに初出判定をリセットする |
| 解答・解説 | 児童用本文とは別に初出判定をリセットする |

題名や前ページに対象語があっても、本文または設問を単独で読んだときに読み支援を失わないようにする。印刷時の改ページ位置は初出判定に影響させない。同じitem revision、DB版、正規化版、ふりがな処理版なら、レイアウト変更後も同じ語にふりがなが付くものとする。

### 初出判定アルゴリズム

表面文字列ではなく、語彙辞書の語句IDで同一語を判定する。同じ漢字表記でも読みや意味が異なる語は別IDとする。活用形は原則として同じ見出し語IDにまとめる。

```text
範囲ごとに seenWordIds を空にする

語を先頭から順に処理する
    ↓
語彙難度が対象学年を超える
    → 別の語へ置き換える
    ↓
grade=1 かつ漢字を含む
    → 語句IDが未出ならふりがな付きで出力し、語句IDを追加する
    → 語句IDが既出ならふりがななしで出力する
    ↓
grade=2|3 かつ漢字表記が既習である
    → ふりがななしで出力する
    ↓
語句IDが seenWordIds にない
    → ふりがな付きで出力し、語句IDを追加する
    ↓
語句IDが seenWordIds にある
    → ふりがななしで出力する
```

各出現箇所には、次の情報を保持する。

```text
語句ID
本文上の表記
読み
文ID
出現順
初出判定の範囲
ふりがなを表示するか
判定理由
```

### 使用量の制約

ふりがなが多すぎると本文が読みにくくなるため、本文長「ふつう」では次を暫定上限とする。

* 初出ふりがな対象は8語以内
* 1文中の新しいふりがな語は2語以内
* 連続する2文の新しいふりがな語は合計3語以内
* 心情推論の根拠文には、推論に不要な新しいふりがな語を置かない

1年生で上限を超える場合は漢字語を減らすか、語をひらがな表記へ置き換える。上限に合わせるために、残した漢字の必須ふりがなを省略してはならない。この上限は固定的な教育基準ではない。複数教材のふりがな密度と、児童が読み止まる箇所を記録して調整する。

### 語句境界の空白

早稲田アカデミーの参照問題12・18で確認した読み支援を参考に、本文・設問・選択肢・解答では、テンプレートが明示した語句または文節相当の境界へ全角空白（U+3000）を1文字分入れる。形態素解析による自動推定は、語を不自然に分割する危険があるため現行プロトタイプでは使用しない。

空白は表示上の読み支援であり、本文文字数、抜き出し文字数、正答照合では除外する。句読点の直前には追加せず、段落と文の境界は別に保持する。境界指定のない任意文へ機械的に空白を挿入してはならない。

## 7.7 心情表現

本文には、次の2種類の心情表現を混在させる。

### 直接的心情表現

```text
うれしくなりました。
心配になりました。
くやしく思いました。
```

### 間接的心情表現

```text
口をむすんで、下を見ました。
何度も時計を見ました。
返事をせずに、くつの先を見つめました。
```

標準4問では、少なくとも問4を、間接表現と出来事から心情を推論する設問にする。問2は明示された心情の理解を測り、問4と同じ構成概念へ重複させない。

## 7.8 本文内の正解配置

設問に対する根拠を、本文中に次のように配置する。

| 設問     | 根拠の置き方               |
| ------ | -------------------- |
| 性格抜き出し | 性格を示す明示語を1か所置く       |
| 心情選択   | 内心または行動を直前・直後に置く     |
| 情報探索   | 明示的な説明を1か所置く         |
| 心情記述   | 発言、行動、状況の複数情報から推論させる |

同じ一文だけですべての設問に答えられる構造は禁止する。

## 7.9 表示と印刷方向

画面表示は操作と校閲をしやすい横書きとする。印刷用紙はA4横・縦書き固定とし、フォームに印刷方向の選択肢を設けない。

- 縦書きでは、児童が記入・読解する導入文、題名、本文、設問、選択肢、解答欄、名前欄に `writing-mode: vertical-rl` を適用する。
- item ID、seed、DB版、状態などの生成情報は児童用問題用紙へ出さず、画面上の「生成根拠・品質」パネルへ集約する。
- 開発用の未校閲注意表示は画面上だけに残し、児童用の印刷物には出さない。人間未確認の問題を児童へ渡さない運用は、印刷物への警告表示ではなくレビュー・リリース状態で制御する。
- 1ページ目を導入文・題名・本文とする。4問セットは2ページ目へ4問を配置し、6問セットは2・3ページ目へ3問ずつ配置する。解答欄の幅を守るために文字を縮めて6問を1ページへ詰め込まない。
- 1ページ目の冒頭には「つぎの　文しょうを　よんで　あとの　もんだいに　こたえましょう」を置き、「文」の初出へ「ぶん」のふりがなを付ける。2ページ目には同義の見出しを重ねない。
- 名前欄は最初の設問ページである2ページ目だけに1か所設け、文字と記入線も縦書き用に配置する。6問セットの3ページ目では重ねて表示しない。
- 縦書き中の設問番号と設問文中の算用数字は正立させ、横倒しにしない。
- 本文ページと各設問ページに同じ外枠・内側余白・左下ページ番号を使い、2枚または3枚を同じ教材セットとして認識できるようにする。
- 設問ページは細い外枠で囲み、右端の先頭列を学年・名前欄、その左を問1から順に配置する。ページ番号は本文ページと同じ左下に置き、独立した設問見出し列は設けない。
- 設問番号には専用領域を確保し、縦書きの設問本文の先頭文字と重ねない。
- 4問セットは実際に必要な記入量に応じて列幅を変える。6問セットは3問ずつ2ページに分け、各ページ内では `answer_layout_id` に応じて列幅を変える。
- `AL_FIXED_CHAR_BOXES` は設問で明示した字数と同数の正方形マスを出す。本文から得た正答長を、字数指定のない設問で暗黙に漏らしてはならない。
- `AL_CHOICE_LIST` は番号を正立した丸囲みとし、選択肢どうしを十分な余白で分ける。各選択肢を長方形で囲まず、別の「一つに○」ラベルも付けない。
- `AL_SINGLE_EXTRACT` は縦方向に連続した無地の抜き出し欄とする。字数指定がないため、1文字単位のマスや補助線を付けず、正答字数を暗黙に示さない。
- `AL_REASON_AND_EMOTION` は「りゆう」と「きもち」の記入領域を分け、前者へ2列、後者へ1列の無地の縦書き記述欄を割り当てる。
- 罫線は本文・設問より薄くし、鉛筆記入を邪魔しない。色に依存せず、白黒印刷でも設問順と記入箇所が分かるようにする。
- 「こたえ」「一つに○」「ぬき出し」の欄種ラベルは付けず、設問文と記入欄そのものだけで回答方法が分かる構成にする。
- アクセント色は設問番号と理由付き記述の採点要素ラベルだけに限定し、本文、設問文、解答マスへは使わない。カラー印刷では視線誘導に使い、白黒印刷でも濃淡差が残る色を選ぶ。
- ふりがな、全角空白、句読点、段落が欠落・重複・はみ出しを起こさないことを印刷PDFの全ページで確認する。
- 印刷処理は本文内容、初出判定、正答、item IDへ影響させない。

## 8. 設問生成仕様

## 8.1 標準構成

[item-blueprint.md](item-blueprint.md) の `standard-reading-4q.v1` に従い、次の4問構成を回帰基準とする。本文構造モジュールは必要な根拠と正規化済み設問内容を提供し、設問セット層が順番・ID・解答欄を確定する。定性参考にした参照問題は `ANCHOR-STORY-Q06` として台帳管理する。

### 問1：人物を表す明示語の抜き出し

測定能力：

* `C1_LOCATE_EXPLICIT`（明示情報の探索）
* 指定条件に従った抜き出し

人物の性格を題材にしていても、本文に明示した語を抜き出させるため、人物像の推論能力を測るとは記録しない。

出題例：

```text
○○は、どのような せいかくでしたか。
文しょう中から 五字で さがして かきぬきましょう。
```

生成条件：

* 正答となる語句を本文中に明示する
* 指定文字数と正答文字数を一致させる
* 同じ文字数の候補が複数存在しないようにする
* 「元気」「やさしい」など、根拠なしでも推測できる語だけにしない

---

### 問2：心情選択問題

測定能力：

* `C2_INTERPRET_EXPLICIT_EMOTION`（明示された心情の理解）
* 場面・人物と明示心情の対応
* 選択肢比較

選択肢は3択を標準とする。

```text
ア：本文と矛盾する心情
イ：一部の状況だけを捉えた誤答
ウ：正答
```

正答位置はランダム化する。

生成条件：

* 正答だけが本文全体と整合する
* 正答の根拠となる心情、願い、期待を本文に直接示す
* 行動だけから心情を導かせる場合は `C3_INFER_EMOTION` へ分類を変え、標準構成として出力しない
* 誤答も日本語として自然にする
* 明らかに幼稚または突飛な誤答を作らない
* 選択肢の長さを極端に変えない
* 正答だけ詳しく書かない
* 正答を示す単語を本文からそのままコピーしすぎない

---

### 問3：明示情報の抜き出し

測定能力：

* `C1_LOCATE_EXPLICIT`（明示情報の探索）
* 問いに対応する箇所の特定

出題例：

```text
○○は、どのような ことが すきですか。
文しょう中から さがして かきぬきましょう。
```

生成条件：

* 正答は本文中に一意に存在する
* 抜き出しの始点と終点が明確
* 不要な修飾語を含むかどうかで採点が曖昧にならない
* 問1と同じ箇所を答えさせない

---

### 問4：心情推論の記述

測定能力：

* `C3_INFER_EMOTION`（心情推論）
* `C5_COMPOSE_WITH_EVIDENCE`（根拠を伴う短い記述）

出題例：

```text
「○○」と ありますが、このときの ○○の 気もちを
かんがえて かきましょう。
```

生成条件：

* 心情を本文に直接書かない
* ただし、推論に必要な根拠は本文中に置く
* 推論は原則1〜2段階とする
* 複数の妥当な表現を許容する
* 模範解答を唯一の言い回しとしない
* 誤字や軽微な表記ゆれを、意味が保たれる範囲で読解得点から分離する
* 記述技能の影響を記録し、児童試行では選択式または口頭回答との比較を検討する

---

## 8.2 設問文の制約

* 一つの設問で二つ以上のことを同時に聞かない
* 「なぜ」と「どんな気持ち」を混在させない
* 指示語の対象を明確にする
* 傍線や番号の参照先を一意にする
* 本文を読まなくても一般常識だけで答えられる問題にしない
* 道徳的に望ましい答えを選ぶだけの問題にしない

## 9. 選択肢生成アルゴリズム

選択肢は、正答1件と誤答2件を生成する。

### 9.1 正答

内部の心情状態から生成する。

例：

```text
内部状態:
自分の工夫が役に立ってうれしい。
```

児童向け表現：

```text
じぶんの くふうが うまくいって、うれしい 気もち。
```

### 9.2 誤答タイプA：逆方向

正答と反対の感情を生成する。

```text
うれしい → かなしい
安心 → 不安
得意 → 恥ずかしい
```

### 9.3 誤答タイプB：部分一致

本文中の別の出来事に対応する感情を選択肢化する。

例：

* 本文前半では不安だった
* 問われている場面では安心している
* 誤答には前半の「不安」を使用する

### 9.4 誤答タイプC：原因の取り違え

感情の方向は近いが、理由を誤らせる。

例：

```text
正答:
友達が手伝ってくれて安心した。

誤答:
先生にほめられて得意になった。
```

### 9.5 誤答品質検査

次の場合は再生成する。

* 本文の解釈によって誤答も正解になり得る
* 正答より誤答の方が本文と整合する
* 誤答が不自然すぎて即座に除外できる
* 選択肢の文体が統一されていない
* 正答だけ文字数が大きく異なる

## 10. 記述問題の採点仕様

## 10.1 採点要素

記述問題は、模範解答との完全一致ではなく、意味要素で採点する。

例：

```text
問:
このときの主人公の気持ちを書きましょう。

必須要素A:
失敗を見られた。

必須要素B:
恥ずかしい、または困った。

任意要素C:
隠したい、逃げたい。
```

### 配点例

| 内容            | 点数 |
| ------------- | -: |
| 心情が適切         | 2点 |
| 原因または状況を捉えている | 2点 |
| 文として意味が通る     | 1点 |

### 許容例

* はずかしかった。
* しっぱいを見られて、はずかしいと思った。
* みんなに見られて、かくれたくなった。
* うまくできなかったことが、くやしかった。

### 不正解例

* うれしかった。
* かなしかったから。
* がんばろうと思った。

ただし、本文の別解釈として成立する回答は、不正解と断定せず部分点対象とする。

## 11. 生成条件の調整

現行プロトタイプで生成難度プロファイルが直接制御するのは、各設問セットの心情記述問題に使う根拠間距離だけである。本文長は独立した入力とする。次は将来の検討候補であり、プロファイル実装済みとはみなさない。

## 11.1 読解負荷を上げ得る要素

* 本文長を「長め」にする
* 登場人物を増やす
* 感情を直接書かず、行動で示す
* 心情変化を2段階にする
* 正答根拠を複数箇所に分散する
* 誤答選択肢を本文の別場面と対応させる
* 抜き出し対象を一文ではなく句にする
* 時間関係を少し複雑にする

## 11.2 生成難度を下げる要素

* 登場人物を1〜2人にする
* 一文を短くする
* 感情語を一部明示する
* 根拠を設問箇所の直後に置く
* 選択肢の差を明確にする
* 抜き出し対象をそのまま本文に置く
* 場面転換を減らす

---

## 12. 品質検査

生成後、次の検査を自動実行する。

## 12.1 本文検査

* 時系列に矛盾がないか
* 登場人物の行動に理由があるか
* 主人公の性格と行動が整合しているか
* 結末が突然すぎないか
* 未回収の出来事が残っていないか
* 語彙が対象学年に適しているか
* 文が長すぎないか
* 参照問題に類似しすぎていないか
* 1年生では漢字を含む全語、2・3年生では未習漢字を含む語の初出に、正しいふりがながあるか
* 同じ範囲の再出語へ、不要なふりがなが繰り返されていないか
* 題名、本文、設問、解答の初出判定範囲が混同されていないか
* 親文字と読み、送り仮名、活用が一致しているか
* 初出ふりがなの密度が設定上限以内か

## 12.2 設問検査

* 各問に正答が一つ以上存在するか
* 選択問題の正答が一つに限定されるか
* 抜き出し問題の文字数が正しいか
* 問いと答えの文法が対応しているか
* 本文だけを根拠に回答できるか
* 同じ能力ばかり測っていないか
* 設問間で答えを教えていないか

## 12.3 内部整合性検査

生成系に、生成した問題を本文だけを使って解かせる。

次の3回の内部判定を行う。

1. 正答生成時の内部情報を使った判定
2. 本文と設問だけを与えた判定
3. 誤答可能性を探す反証判定

3つの判定が一致しない場合は、問題を修正または破棄する。

これらは同じ生成系または同じ実装上の前提を共有し得るため、別観点の見直しとは呼ばない。合格しても `automated_checks_passed` までであり、答えを隠した別セッションでの解き直しを代替しない。

## 12.4 テンプレート再利用検査

`template_id` が指定された生成では、通常の品質検査に加えて次を確認する。

* テンプレート状態が `approved_template` である
* テンプレートの `blueprint_id`、学年範囲、本文長、難度条件が入力と矛盾しない
* 抽象スロットが具体的な人物・舞台・出来事へ一意に展開される
* 元の採用itemの不変条件（正答根拠、問ごとの測定能力、許容距離）が維持される
* 過去の同一テンプレート由来itemと、設定した類似度・使用回数の上限を超えない
* `template_id`、`template_revision`、テンプレートハッシュが生成来歴に残る

テンプレートから展開したitemも新しいitem revisionとして扱い、元テンプレートのレビュー状態や実測値を自動継承しない。本文、設問、語彙、根拠を変更した場合は、そのitemについて通常のレビュー・校正をやり直す。

## 13. 自動品質スコア

各問題セットを100点満点で評価する。

| 評価項目     | 配点 |
| -------- | -: |
| 本文の自然さ   | 15 |
| 因果関係の明確さ | 15 |
| 学年適合性    | 15 |
| 設問の一意性   | 20 |
| 心情推論の妥当性 | 15 |
| 選択肢の品質   | 10 |
| 類似回避     | 10 |

合計85点以上を、未校閲の練習素材として出力可能とする。

70〜84点は再編集する。

69点以下は破棄して再生成する。

この点数は生成系内部のスクリーニング値であり、測定妥当性、開発者による見直し、実測難易度、公平性、公開可否を保証しない。`release_ready` への状態遷移には使用しない。

## 14. 生成処理フロー

```text
1. 入力条件を受け取る
        ↓
2. generator・仕様・DB・テンプレート・モデル等の版を固定する
        ↓
3. 乱数シードを確定する
        ↓
4. アルゴリズム生成はseedから出題設計IDを決定的に選び、AI生成は設計図に対応するIDを選んで、登録済み本文構造モジュールを解決する
        ↓
5. template_idが指定されていれば承認済みテンプレートを読み込む
   指定されていなければ題材カテゴリ・主人公・舞台・性格・目的を抽選する
        ↓
6. 選択した本文構造モジュールのscenario契約に従い、中心事件と心情変化を生成する
        ↓
7. 因果関係を検査する
        ↓
8. 正答と根拠箇所を先に作る
        ↓
9. 本文構造モジュールで、根拠を含む役割列を作る
        ↓
10. 児童向けの本文に文章化する
        ↓
11. 語彙難度と表記難度を分けて判定する
        ↓
12. 範囲ごとの初出ふりがな計画を作る
        ↓
13. 同じ本文構造モジュールで、選択肢と記述問題を生成する
        ↓
14. 模範解答と採点基準を生成する
        ↓
15. 学年・表記・ふりがな・文字数を検査する
        ↓
16. 共通検査と本文構造固有検査で、正答一意性と根拠契約を検査する
        ↓
17. 参照アンカーとの表層類似度を検査する
        ↓
18. 自動品質スコアを算出する
        ↓
19. 基準未達なら再生成する
        ↓
20. 現行プロトタイプでは、生成来歴、`lifecycle_status=automated_checks_passed`、`usage_class=development_preview`、`child_use_permitted=false` を付けて出力する
```

固定問題を見直し済み・校正済みとして採用する場合は、出力後に [item-review-and-release.md](item-review-and-release.md) の状態遷移へ進む。

## 17. 生成品質コーパス検査

ランダム生成能力を検査する場合、1問だけでは判定しない。

学年と生成難度プロファイルの各組合せについて、同一条件・異なる乱数シードで最低30セットを生成する。

### 評価項目

* 題材の重複率
* 主人公カテゴリの分布
* 舞台の分布
* 中心事件の分布
* 心情変化の分布
* 文章構造の重複率
* 正答不能問題の発生率
* 複数正答問題の発生率
* 大人による自然さ評価

### 目標値

| 指標           |     目標 |
| ------------ | -----: |
| 正答不能率        |   1％未満 |
| 複数正答率        |   3％未満 |
| 題材完全重複率      |   5％未満 |
| 文章骨格の高類似率    |  10％未満 |
| 学年不適合語の発生率   |   2％未満 |
| 成人評価者による自然判定 |  90％以上 |

30セット生成はシステムの多様性・不良率を調べる検査であり、児童にとっての難しさや識別力を示さない。対象児童数、実施条件、正答率、無回答率、選択肢分布、再採点一致、公平性の評価は [calibration-and-fairness.md](calibration-and-fairness.md) に従う。

## 18. 非機能要件

### 再現性

完全に同じ問題を再生成するには、少なくとも次を固定する。

- generator版
- アルゴリズム仕様版
- 出題設計版
- DB版
- seed
- テンプレート・プロンプト版
- 使用モデルIDと推論設定
- 正規化・ふりがな処理版

これらを固定しても外部モデルが決定的でない場合は、完全再生成を保証せず、生成済み本文・設問・解答をitem revisionとして保存する。

「同じ問題を再生成できること」と「同じ出題設計で同等問題を新しく生成できること」を別の要件・テストとして扱う。

### 多様性

同じ本文構造モジュール内では、乱数シードを変更した場合に題材、人物、表層的な出来事、補助文、選択肢位置が変化すること。構造間の多様性は、seed群で `story-retry-craft.v1` と `story-clue-discovery.v1` の両方が生成され、各構造固有の契約検査を通ることを確認する。ただし、2種類あることだけで十分な多様性や文章品質を達成したとはみなさず、生成分布と人間レビューを別に記録する。

### 説明可能性

各正答について、本文のどこを根拠としたか確認できること。

### 修正可能性

本文、設問、選択肢、生成難度を個別に再生成できること。

一部を再生成した場合は新しいitem revisionとし、旧版の見直し・校正状態をそのまま引き継がない。

### 安全性

次の題材は標準設定では生成しない。

* 児童への深刻な暴力
* 死亡
* 虐待
* 強い恐怖
* 差別的表現
* 犯罪の具体的手順
* 年齢に不適切な恋愛・性的内容
* 実在人物への中傷
* 家庭環境を決めつける表現

禁止題材に該当しないだけで公平とみなさない。家庭構成、経済環境、地域・文化経験、性別役割、障害・発達特性、表示アクセシビリティは専用チェックリストと児童試行で確認する。

## 19. 実装上の重要原則

本システムの価値は、文章を大量に生成することではない。

次の4点を同時に満たすことにある。

1. 題材と物語は十分に多様である
2. 測定対象とする読解能力が出題設計で固定されている
3. 正答と根拠が機械的に検証できる
4. 機械検査、別セッションでの見直し、児童試行を区別できる

したがって、生成処理の中心は文章生成モデルではなく、次の構造管理に置く。

```text
出題設計
→ 本文構造モジュール
→ 正答
→ 根拠
→ 物語構造
→ 本文表現
```

本文構造モジュールはジェネレータ固有ディレクトリ内に置き、各モジュールが題名、本文、設問、正答根拠、構造固有検査を一体として提供する。複数教材で共通利用する実績がない段階で、リポジトリ共通の汎用ライブラリへ昇格しない。

「本文を作り、あとから問題を考える」という順序にはしない。

## 20. 出力・完成条件

### 20.1 未校閲の練習素材として出力する条件

次の条件をすべて満たした問題だけを `unreviewed_practice` として出力する。

現行プロトタイプは正式語彙DB、語彙の学年適合性、人間レビューを満たしていないため、この状態へ進めず `development_preview` とする。

* 本文が対象学年に適している
* 本文だけで全問に答えられる
* 抜き出し問題の正答が一意である
* 選択問題の正答が一つである
* 記述問題に明確な採点要素がある
* 設問構成と測定能力が出題設計IDに一致する
* 心情推論に本文上の根拠がある
* 過去問題と物語骨格が重複していない
* 道徳的な常識だけでは答えられない
* 自動品質スコアが85点以上である
* 語彙難度が対象学年に適している
* 1年生では漢字を含む全語、2・3年生では未習漢字を含む語に、定義された範囲の初出で正しいふりがなが付いている
* 同じ範囲の再出語では、意図どおりふりがなが省かれている
* ふりがなの親文字、読み、送り仮名、活用に不整合がない
* 初出ふりがなの密度が設定上限を超えていない
* 必須の生成来歴、item ID、revision、利用状態を保持している
* 未校閲・未校正であり学力判定に使えないことを利用者へ表示する

### 20.2 見直し済み・校正済みとして完成する条件

次をすべて満たした固定問題版だけを `release_ready` とする。

* 作問チェックを完了している
* 正答と根拠を隠した別セッションで解き直している
* 表現・公平性・アクセシビリティ確認に合格している
* 対象学年・実施条件を固定した児童試行を完了している
* 実測難易度、無回答率、誤答分布、識別力を記録している
* 記述問題の再採点一致が基準を満たしている
* 最終編集とは別のセッションで公開条件を確認している
* 修正時に見直し・校正状態を必要な地点まで戻せる

詳細な状態遷移は [item-review-and-release.md](item-review-and-release.md)、数値基準と公平性分析は [calibration-and-fairness.md](calibration-and-fairness.md) を正本とする。
