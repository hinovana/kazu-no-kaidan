# こくごのたね 出題・測定設計仕様書

| 項目 | 値 |
| --- | --- |
| 文書状態 | 検討中 |
| 文書版 | `item-blueprint.v0.6-draft` |
| 開発形態 | 個人開発 |
| 最終更新日 | 2026-07-20 |
| 実装状態 | 本文構造3種類と、本文構造から分離した設問セット3種類を実装。人物状態・関係状態・伏線回収の測定契約を採用したが、根拠グラフと対応設問は未実装 |
| 実証状態 | 未実証・未校正 |
| 現段階の実装範囲 | 標準4問・因果6問・文脈と心情4問を生成する未校閲練習用プロトタイプまで |
| 主な未決事項 | 必要に応じた専門家への相談、説明文設計、児童試行による妥当性 |

現行プロトタイプは3種類の設問セットを生成するが、測定能力や配点の妥当性を実証したものではない。最初の人間レビューでは、実際の生成画面を見ながら各設問が意図した構成概念を問えているか、設問セットの違いが本文根拠と整合するかを確認する。本文構造や設問セットの種類が多いこと自体を文章品質や出題妥当性の証拠とはしない。

## 1. 目的

本書は、国語読解問題ジェネレータ「こくごのたね」が、何を測り、何を測らず、どのような本文上の反応を根拠に得点を与えるかを定める。

文章や設問を自然に生成できることと、狙った読解能力を測定できることは別である。作問アルゴリズムは、先に本書の出題設計を選び、その制約を満たす本文・設問・正答根拠を生成しなければならない。

## 2. 初版の適用範囲

初版の対象は、小学1〜3年生向けの短い**物語文**に限定する。

説明文は、中心文、具体例、順序、比較、因果、指示語、要旨など、物語文とは異なる測定設計を必要とする。説明文専用の出題設計を確定するまで、UI、生成API、テストデータの選択肢へ `expository` を公開しない。

本書の対象外は次のとおり。

- 説明文、詩、随筆、古典
- 漢字の書き取りや語彙知識そのものを採点する問題
- 読書速度を測る問題
- 学力診断や合否判定に単独で使用する標準化テスト

### 2.1 出題設計と本文構造モジュール

現行実装では、本文構造と設問構成を別のIDで追跡する。`blueprint_id` は本文構造モジュールの互換性を含む既存の出題設計ID、`story_structure_id` は本文の役割列、`question_set_blueprint_id` は1枚に並べる設問パターンと順番を表す。各問は `question_pattern_id` と `answer_layout_id` を持つ。

本文構造モジュールは、題名、本文の役割列、根拠配置と、設問パターンが利用する正規化済み `QuestionContent` を提供する。完成した設問の順番、`question_id`、解答欄は設問セット層が決める。本文構造だけを差し替えて根拠契約を従来のまま流用したり、必要な根拠がない設問パターンを後付けしたりしてはならない。

本文構造モジュールは少なくとも次を宣言する。

```text
blueprint_id
story_structure_id
text_type
anchor_ids
題名生成規則
入力材料から本文用scenarioへ変換する規則
本文の役割列と順序
設問パターンに渡すQuestionContentと正答根拠
構造固有の機械検査
```

物語状態・根拠グラフ拡張の段階1以降では、これに `EvidenceGraphDraft` を加える。`EvidenceGraphDraft` は本文構造モジュール内の文の役割を参照し、本文組立て後に生成エンジンが `SentenceId` へ解決した `EvidenceGraph` を作る。人物状態、関係状態、伏線回収は、完成した本文から後付けで推測せず、このグラフ下書きへ明示する。現在のコードは未対応である。

現行実装は次の3組を登録する。アルゴリズム生成でIDを明示しない場合は、seedから決定的に1組を選ぶ。同じ条件とseedでは、本文構造を含む生成結果が変わらない。

| `blueprint_id` | `story_structure_id` | 構造 | 状態 |
| --- | --- | --- | --- |
| `story-standard-4q.v1` | `story-retry-craft.v1` | 作成開始、期待、小さな失敗、確認方法の決定、失敗を見られた反応、修正、解決 | 実装済み・未校閲 |
| `story-clue-discovery-4q.v1` | `story-clue-discovery.v1` | 観察開始、期待、手がかりの発見、確かめ方の決定、手がかりの一致、うなずく反応、発見の説明 | 実装済み・未校閲 |
| `story-late-arrival-4q.v1` | `story-late-arrival.v1` | 二人で作業開始、小さな問題、第三人物の途中参加、第三人物の介入、主人公による方法の決定、三人で解決 | 実装済み・未校閲 |

本文構造とは独立に、次の設問セットをseedから決定的に選ぶ。内部IDはフォームへ公開せず、生成来歴で確認できるようにする。

| `question_set_blueprint_id` | 問数 | 構成 | 状態 |
| --- | ---: | --- | --- |
| `standard-reading-4q.v1` | 4 | 明示語、明示心情、明示事実、心情記述 | 実装済み・未校閲 |
| `causal-trace-6q.v1` | 6 | 原因、明示心情、明示事実、文脈、心情記述、結果 | 実装済み・未校閲 |
| `context-and-inference-4q.v1` | 4 | 行動根拠型心情、結果、文脈、心情記述 | 実装済み・未校閲 |

`story-plan.v1` は失敗・確認・やり直しを前提とするため、AI生成では現時点で `story-standard-4q.v1` だけを使う。手がかり発見型と途中参加型はローカル生成専用であり、`story-plan.v1` との組合せを拒否する。これらをAI生成へ接続するときは、対応する設計図スキーマまたは明示的な正規化アダプターを別途追加する。今後さらに比較、協働などを加える場合も、既存モジュールへ条件分岐を積み重ねず、独立した本文構造モジュールと契約テストを追加する。

## 3. 測定する能力

### 3.1 構成概念コード

| コード | 能力 | 操作的定義 | 正答を示す反応 | 測らないもの |
| --- | --- | --- | --- | --- |
| `C1_LOCATE_EXPLICIT` | 明示情報の探索 | 設問条件に合う、本文中に明記された語句または事実を特定する | 指定範囲から正しい語句を抜き出す | 背景知識による補完、人物像の推論 |
| `C2_INTERPRET_EXPLICIT_EMOTION` | 明示された心情の理解 | 本文に直接示された心情・願い・期待を、場面と人物に対応付ける | 明示表現と一致する選択肢を選ぶ | 行動から心情を推測すること |
| `C3_INFER_EMOTION` | 心情推論 | 出来事、発言、行動のうち指定された根拠を結び付け、本文に直接書かれていない心情を導く | 根拠と矛盾しない心情を選択または記述する | 道徳的に望ましい答え、一般常識だけの推測 |
| `C4_INFER_TRAIT` | 人物像の推論 | 複数の行動・選択から、人物の比較的持続する特徴を導く | 2つ以上の行動根拠に合う人物像を答える | 本文に書かれた性格語の単純な抜き出し |
| `C5_COMPOSE_WITH_EVIDENCE` | 根拠を伴う短い記述 | 本文上の出来事と心情・判断を、指定された1〜3文で結び付ける | 採点要素を満たす因果関係のある記述をする | 字の巧拙、表記の完全性、作文の華やかさ |
| `C6_CONNECT_CAUSE_RESULT` | 原因と結果の対応 | 本文に示された原因と、その後の行動・結果を対応付ける | 本文の因果関係と一致する選択肢を選ぶ | 一般常識だけによる原因推測 |
| `C7_INTEGRATE_CONTEXT` | 前後文脈の統合 | 出来事の前後関係を読み、指定場面に続く行動を特定する | 前後の両方と整合する行動を選ぶ | 単語の表面的一致だけによる選択 |
| `C8_RESOLVE_REFERENCE` | 指示内容の解決 | 指示語・抽象表現を、本文中の一意な出来事要素へ対応付ける | 必要な人物・行動・結果を本文に即して説明する | 表面上近い単語だけの照合、背景知識による補完 |
| `C9_TRACK_CHARACTER_STATE` | 人物状態の変化追跡 | 離れた場面の感情、知識、意図、他者評価を同一人物へ対応付け、変化とその契機を捉える | 変化前後と本文上の契機を正しく結び付ける | 一場面だけの明示心情の再認、道徳的評価 |
| `C10_TRACK_RELATIONSHIP_STATE` | 関係状態の変化追跡 | 二人以上の関係について、初期状態、変化を起こす行動、相手の応答、変化後の状態を結び付ける | 関係の悪化・維持・修復を本文中の相互作用から説明する | 片方の心情だけによる推測、仲直りを善とする価値判断 |
| `C11_CONNECT_CALLBACK` | 伏線と回収の統合 | 一つ以上の意味段階を隔てた序盤の期待・約束・助言・未完了情報と、終盤の合図・想起・発見・行動変化を対応付ける | 序盤の設定が終盤の感情または行動へどう関与したかを答える | 隣接文の単純な順序判断、直接隣接する一段階因果 |

### 3.2 境界を混同しない

本文に「あわてんぼう」と明記し、それを抜き出させる設問は、人物像を題材としていても主に `C1_LOCATE_EXPLICIT` を測る。これを `C4_INFER_TRAIT` と記録してはならない。

本文に「かなしい」と明記されている場合、その語を場面へ対応付ける設問は `C2_INTERPRET_EXPLICIT_EMOTION` である。伏し目、沈黙、行動の変化などから心情を導く場合だけ `C3_INFER_EMOTION` とする。

記述問題は、読解だけでなく文の組み立てや表記の影響を受ける。`C5_COMPOSE_WITH_EVIDENCE` の得点を読解能力だけの得点として扱わず、児童試行では選択式または口頭回答との比較を行う。

### 3.3 物語状態を測定対象にする範囲

人物状態、関係状態、伏線回収は、本文へその要素が存在するだけでは測定済みとしない。各設問が要求する認知操作と正答根拠によって、次のように扱う。

- 一場面の明示心情を選ぶだけなら `C2_INTERPRET_EXPLICIT_EMOTION` とし、`C9_TRACK_CHARACTER_STATE` にしない。
- 行動や反応から一場面の心情を推論するだけなら `C3_INFER_EMOTION` とし、変化前後を追わない限り `C9_TRACK_CHARACTER_STATE` にしない。
- 人物状態は、同じ `character_id` の異なる `phase_id` に属する状態ノードと、その変化を生じさせた根拠を使う場合だけ測定対象にする。
- 関係状態は、二人以上の `character_id`、変化を生じさせた出来事、相手の応答、変化後の状態を使う場合だけ `C10_TRACK_RELATIONSHIP_STATE` とする。
- 伏線回収は、設定ノードと回収ノードの間に一つ以上の意味段階があり、両方を読まなければ答えられない場合だけ `C11_CONNECT_CALLBACK` とする。隣接する原因と結果は `C6_CONNECT_CAUSE_RESULT`、直前直後の整合は `C7_INTEGRATE_CONTEXT` とする。
- 指示語・抽象表現の説明は `C8_RESOLVE_REFERENCE` とし、人物状態や関係状態を答えに含んでも、主たる操作が参照先の特定なら構成概念を置き換えない。

物語状態・根拠グラフはこれらを検査するための生成基盤であり、グラフを保持していること自体を児童の能力測定とはみなさない。`C8`〜`C11` は採用済み・未実装であり、対応する設問パターンと本文構造が実装されるまで生成物へ付与しない。

## 4. 測定しない要因と統制

正答率へ次の要因が不要に混入しないようにする。

| 要因 | 統制方法 |
| --- | --- |
| 漢字の読み | 1年生では全漢字、2・3年生では未習漢字について、学年・学期プロファイルと初出ふりがな規則を適用する |
| 学年外語彙 | 語彙DBで除外し、ふりがなだけで解決しない |
| 家庭・地域・文化経験 | 本文中に必要情報を与え、経験者だけが有利になる設問を作らない |
| 道徳判断 | 「よい行動」を選ばせず、本文の人物が実際に感じたことを問う |
| 記憶負荷 | 根拠箇所を本文へ残し、本文を見ながら回答できるようにする |
| 筆記技能 | 表記ゆれや軽微な誤字を、意味が保たれる範囲で読解得点から分離する |
| 読む速さ | 初版では制限時間による得点差を付けない |
| 設問テクニック | 他の設問や選択肢が正答を教えないようにする |

## 5. 標準4問の出題設計

`standard-reading-4q.v1` は回帰基準となる4問構成を次のように固定する。他の設問セットは [question-pattern-expansion.md](question-pattern-expansion.md) の採用済み範囲に従う。

| 順番 | 設問コード | 主たる測定能力 | 形式 | 必要な本文根拠 | 標準配点 |
| --- | --- | --- | --- | --- | ---: |
| 1 | `Q1_TRAIT_TERM_EXTRACT` | `C1_LOCATE_EXPLICIT` | 指定字数の抜き出し | 人物を表す明示語1か所 | 1 |
| 2 | `Q2_EXPLICIT_EMOTION_CHOICE` | `C2_INTERPRET_EXPLICIT_EMOTION` | 3択 | 直接的心情表現または願い1か所 | 1 |
| 3 | `Q3_EXPLICIT_FACT_EXTRACT` | `C1_LOCATE_EXPLICIT` | 抜き出し | 出来事・理由の明示箇所1か所 | 1 |
| 4 | `Q4_INFER_EMOTION_RESPONSE` | `C3_INFER_EMOTION` と `C5_COMPOSE_WITH_EVIDENCE` | 1〜3文の記述 | 状況と反応を含む2か所以上 | 2 |

問1は人物像に関する語句を扱うが、明示語の抜き出しであるため、人物像の推論問題とは呼ばない。`C4_INFER_TRAIT` を出題する場合は、標準4問の差し替え版として別の設計IDを付け、複数の行動根拠を必須にする。

問2の根拠を間接表現へ変更した場合、測定能力は `C3_INFER_EMOTION` になる。問4と同じ能力が重複するため、標準構成としては不採用とし、別の出題設計として版管理する。

## 6. 正答根拠の要件

各設問は、生成前に次を確定する。

```text
blueprint_id
question_set_blueprint_id
question_id
question_pattern_id
answer_layout_id
primary_construct
secondary_demands
answer
acceptable_answers
evidence_ids
required_inference_steps
scoring_elements
disqualifying_answers
```

正答根拠は次の条件を満たす。

- 本文だけから到達できる。
- 正答に必要な情報と、もっともらしい誤答の根拠を区別できる。
- 抜き出し問題では、指定条件を満たす候補が1つである。
- 選択問題では、正答が1つであり、誤答は本文の別情報または推論上の誤りに対応する。
- 推論問題では、推論段階と使用する根拠を列挙できる。
- 背景知識を使わなくても正答できる。

物語状態・根拠グラフへ依存する設問は、上記に加えて `evidence_node_ids` を持ち、必要に応じて `evidence_edge_ids`、`evidence_collection_id`、`story_callback_id` のいずれかを持つ。これらは `SentenceId` を置き換えるものではなく、意味上の根拠から本文箇所まで追跡するために併記する。現行8パターンは段階1の互換アダプターが従来の `evidence_ids` を供給するため、設問契約を同時に変更しない。

## 7. 許容される変形範囲

同じ出題設計IDのまま変更できる要素は次のとおり。

- 登場人物の名前、性別を固定しない属性、人数の範囲
- 学校、家庭、町、自然などの舞台
- 出来事の表層、道具、季節
- 直接話法と地の文の比率
- 学年プロファイルに応じた語彙・漢字・ふりがな
- 同じ `story_structure_id` が定める役割列と根拠契約、および同じ設問パターンへ渡すQuestionContentを保った表層表現

次を変更した場合は同じ出題設計IDを使わず、新しい設計版または別設計にする。

- 主たる測定能力
- 正答に必要な推論段階数
- 1根拠か複数根拠か
- 選択式、抜き出し、記述の変更
- 採点要素
- 物語文から説明文への変更
- 設問セットの構成比率

本文構造の役割列や中心となる因果の型を変更する場合は、少なくとも新しい `story_structure_id` を付与する。変更によって根拠配置、推論段階、設問形式または採点要素が変わる場合は、新しい `blueprint_id` も付与する。

段階1で現行3構造へ `EvidenceGraphDraft` を追加する変更は、既存の役割列、題名、本文、設問、正答、解答欄を変えない内部契約の追加として扱い、現行の `story_structure_id`、`blueprint_id`、`question_pattern_id`、`question_set_blueprint_id` を維持する。新しい人物状態の並び、関係状態の遷移、伏線回収を本文の必須構造として追加する場合は、新しい `story_structure_id` と `blueprint_id` を付与する。

根拠グラフを生成来歴へ追加する最初の実装では、generator版を `kokugo-no-tane.prototype.v0.11`、根拠グラフ版を `evidence-graph.v1` とする。児童向けの題名、本文、設問文、選択肢順、正答、解答欄、印刷ページは現行seedで維持し、生成来歴と開発者向け診断の追加だけを意図した差分とする。旧生成物へグラフを捏造して同じrevisionのまま追記せず、必要な場合は再生成した新しいitem revisionとして扱う。

参照問題から再利用してよいのは、測定能力、根拠配置、設問形式、読み支援などの抽象的特徴だけである。人物、固有の出来事、文章、選択肢、展開順を模倣しない。参照根拠は [reference-anchor-registry.md](reference-anchor-registry.md) のアンカーIDで記録する。

## 8. 学年と生成難度

### 8.1 学年

学年は、語彙、漢字、ふりがな、文法表現、題材経験の上限を制御する。学年を上げただけで、推論段階や根拠間距離を増やさない。

### 8.2 生成難度プロファイル

現行の1〜5は実測難易度ではなく、各設問セットに1問含まれる心情記述問題の2つの根拠文をどれだけ離すかを指定する生成条件である。

| プロファイル | 心情記述問題の根拠配置 |
| --- | --- |
| 1 | 隣り合う文 |
| 2 | 隣り合う文、同一段落 |
| 3 | 1文以上を挟む（文距離2以上） |
| 4 | 文距離3以上、異なる段落 |
| 5 | 文距離5以上、異なる段落 |

現行プロトタイプでは、人物数、推論段階、心情表現、選択肢の近さをプロファイルごとに変えていない。人物数はプロファイルから独立した本文構造の契約として、二人固定の2構造と、途中から第三人物が加わる `story-late-arrival.v1` をseedで選ぶ。人物数以外の要素を今後実装する場合も、何を測定する設問かを崩さないことを確認し、別の生成条件として追加する。

### 8.3 本文長

本文長は生成難度と独立して選択する。全角空白とふりがなの読みを除いた文字数帯は次のとおりとする。

| 本文長 | 文字数帯 |
| --- | ---: |
| 短め | 250〜370字 |
| ふつう（初期値） | 370〜570字 |
| 長め | 500〜750字 |

実測難易度の付与は [calibration-and-fairness.md](calibration-and-fairness.md) に従う。

## 9. 題材と公平性

題材は、次を満たすものだけを使用する。

- 本文中の説明だけで状況を理解できる。
- 特定の家庭構成、経済環境、居住地域、文化行事の経験を前提にしない。
- 性別と性格・役割・得意不得意を固定的に結び付けない。
- 障害、病気、国籍、家庭事情を、驚きや失敗のためだけの装置にしない。
- 正答するために、スポーツやゲームなどの専門ルールを知っている必要がない。
- 児童が不安や恐怖を強く感じる内容を標準設定で扱わない。

自動検査だけで公平性を保証したとみなさない。開発者本人の専用チェックと児童試行は [item-review-and-release.md](item-review-and-release.md) と [calibration-and-fairness.md](calibration-and-fairness.md) に従う。

## 11. 完成条件

出題設計は、次をすべて満たすまで確定扱いにしない。

- 測定する能力と測定しない能力が操作的に定義されている。
- 各設問に主たる測定能力が1つ定義されている。
- 心情理解と心情推論、明示語の抜き出しと人物像推論を区別している。
- 学年と生成難度を独立に指定できる。
- 背景知識、語彙、筆記技能の影響を抑える規則がある。
- 許容される変形と、別設計にすべき変更が区別されている。
- 本文構造モジュールが題名、本文、根拠、設問、構造固有検査を一体として提供し、生成来歴で識別できる。
- 人物状態、関係状態、伏線回収を測る場合と、単一場面の心情・因果・順序を測る場合の境界が定義されている。
- 物語文以外を未定義のまま生成対象にしていない。
- 参照アンカーの出典と利用範囲を追跡できる。
- 見直し、校正、公平性の正本へ接続している。
