| 項目 | 値 |

本書は、文章生成モデルを「こくごのたね」へ接続する場合の責務、セキュリティ、データ形式、検証ゲート、段階的な導入手順を定める。現在の決定的テンプレート生成を置き換える文書ではない。

本文・設問・正答根拠の生成規則は [algorithm-draft.md](algorithm-draft.md)、測定対象は [item-blueprint.md](item-blueprint.md)、ローカルライブ接続のHTTP・UI・起動契約は [local-ai-proxy-spec.md](local-ai-proxy-spec.md)、公開状態と人間レビューは [item-review-and-release.md](item-review-and-release.md) を正本候補とする。

## 1. 結論

初期導入では、文章生成モデルを**完成問題の作者**ではなく、**物語設計図の候補供給者**として使用する。

```text
学年・カテゴリ・本文長・生成条件
        ↓
文章生成モデルが物語設計図候補をJSONで生成
        ↓
スキーマ・内容・語彙・類似・安全性を検査
        ↓
選択された本文構造モジュールと設問セットが本文・4問または6問・正答根拠を決定的に生成
        ↓
既存の機械検査と人間レビュー
        ↓
合格した固定データだけをブラウザへ配布
```

文章生成モデルの出力を、そのまま児童向け画面へ表示してはならない。モデル出力は、外部から受け取った未検証データとして扱う。

## 2. 導入目的

- カテゴリごとの舞台、人物、目的、出来事、失敗、解決方法を増やす。
- 手書きテンプレートだけでは得にくい組合せ候補を効率よく収集する。
- 現在の3設問セット、正答根拠、初出ふりがな、語句間隔、文字数、印刷仕様を維持する。
- 良問候補の母数を増やし、人間レビューで比較できるようにする。

「ほぼ無限に異なる文章を出すこと」自体を品質目標にしない。目標は、同じ測定設計を保ったまま、自然で年齢相応な良問候補を十分に得ることである。

## 3. 初期導入：事前生成型

開発時にNode.jsスクリプトからAPIへ接続し、候補を静的JSONへ変換する。児童がブラウザで問題を生成するときには外部APIへ接続しない。

```text
scripts/generate-story-plans.mjs
        ↓ server-side API key
OpenAI Responses API
        ↓ Structured Outputs
data/model-candidates/*.json
        ↓ import・normalize・validate
data/normalized/story-plans.json
        ↓ build
data/generated/story-plan-pack.json
        ↓
ブラウザ内の決定的ジェネレータ
```

この方式では、次を維持できる。

- APIキーを配布物へ含めない。
- 利用時の通信、待ち時間、API障害、利用料金を発生させない。
- モデル出力を事前に保存し、同一問題を完全再現する。
- 機械検査を通過した候補だけをブラウザへ渡す。
- 子どもの名前、回答、操作履歴をモデルへ送らない。

大量の候補を即時性なしで生成する段階では、非同期の [Batch API](https://developers.openai.com/api/docs/guides/batch) を検討できる。ただし、通常のResponses APIで小規模にスキーマと評価方法を固めてから移行する。

## 4. 開発用に実装したライブ生成型

ブラウザ操作ごとに新しい設計図を生成する開発モードとして、ブラウザとAI providerの間にNode.js製ローカルプロキシを実装した。評価中はCodex CLIのヘッドレス実行を既定とし、OpenAI Responses APIは環境変数で明示的に選ぶ互換経路とする。HTTP・CORS・UI・起動契約は [local-ai-proxy-spec.md](local-ai-proxy-spec.md) に従う。

```text
ブラウザ
  ↓ 学年・生成条件プロファイル・カテゴリ・本文長・seedを送る
自前の /api/story-plan
  ↓ KNT_AI_PROVIDERでサーバー側だけを切り替える
Codex CLIヘッドレス または OpenAI Responses API
  ↓ JSON候補
サーバー側検査・保存・キャッシュ
  ↓ 合格候補だけを返す
ブラウザ側の決定的ジェネレータ
```

Codex経路は既存のCodexログインを使い、APIキーを要求しない。OpenAI API経路を選ぶ場合も、公式APIリファレンスに従い、APIキーをブラウザやアプリのクライアントコードへ公開せずサーバー環境変数から読む。現在の静的ブラウザアプリへprovider選択やAPIキーを埋め込む方式は採用しない。

ライブ生成を採用する場合は、少なくとも次を追加する。

- 呼出回数とトークン量の上限
- タイムアウト、限定回数の再試行、ローカル生成へのフォールバック
- 入力値のallowlist検証
- 同一条件のキャッシュ
- APIリクエストID、失敗理由、利用量の記録
- 不正利用と大量呼出しへの制限
- モデル停止時にも既存の問題生成を継続できる構成

開発者PCでNode.jsプロキシを `http://127.0.0.1:8787` に起動する場合の起動コマンド、環境変数、CORS、`GET /health`、`POST /api/story-plan`、ブラウザUI、エラー、候補保存、テスト、完了条件は [local-ai-proxy-spec.md](local-ai-proxy-spec.md) を正本候補とする。

## 5. Providerと出力形式

Codex経路では `codex exec --output-schema`、API経路では [Responses API](https://developers.openai.com/api/docs/guides/text) の [Structured Outputs](https://developers.openai.com/api/docs/guides/structured-outputs) を使う。どちらも出力を自由記述JSONにせず、同じJSON Schemaで制約する。

Structured Outputsは形式と必須キーを保証するが、教育的な正しさ、自然さ、学年適合性、正答一意性までは保証しない。既存の検証処理を省略してはならない。

### 5.1 物語設計図の最小例

現行の `story-plan.v1` は `story-standard-4q.v1`／`story-retry-craft.v1` 専用の入力契約である。小さな失敗、確認行動、やり直し、解決を必須とするため、発見、比較、協働など別の本文構造へそのまま流用しない。新しい本文構造を追加する場合は、そのモジュールが受理する設計図スキーマまたは正規化アダプターを明示し、未対応の組合せを生成前に拒否する。

```json
{
  "schema_version": "story-plan.v1",
  "category": "町",
  "title_concept": "まちたんけんのちず",
  "setting": {
    "type": "public_space",
    "name": "まちのひろば"
  },
  "protagonist": {
    "name": "あお",
    "role": "しょうがくせい"
  },
  "supporting_character": {
    "name": "ゆう",
    "role": "ともだち"
  },
  "goal": "まちたんけんのちずをかんせいさせる",
  "event": {
    "action": "めじるしをかこう",
    "problem": "めじるしをちがうばしょにかいてしまいました",
    "decision": "あるいたじゅんばんをたしかめること",
    "resolution": "ただしいばしょへかきなおしました"
  },
  "emotion": {
    "before": "うれしい",
    "after": "はずかしい"
  },
  "evidence_requirements": [
    "せいかくをしめすことばをおける",
    "しっぱいをみられたばめんとはんのうをわけられる"
  ]
}
```

実際のJSON Schemaでは、全オブジェクトの必須項目、文字数上限、enum、配列件数、追加プロパティの禁止を定義する。`domain/schemas/story-plan-v1.ts` をブラウザとサーバーで共有し、JSON Schema、TypeScript型、追加の内容検証を同じモジュールで管理する。

### 5.2 モデルに生成させないもの

初期導入では、次をモデルへ委ねない。

- 最終的な設問文
- 正答と採点基準
- 本文根拠ID
- 既習漢字判定
- 初出ふりがな判定
- 抜き出し文字数
- 心情記述問題の根拠間距離
- 公開可否または品質合格判定

これらは既存の仕様、DB、決定的アルゴリズム、検証コードが決定する。

## 6. 入力プロンプトの契約

アプリ側の固定指示と、生成条件を分離する。

### 固定指示

- 小学1〜3年生向けの物語設計図候補だけを生成する。
- 文章本文、設問、解答は生成しない。
- 指定したJSON Schemaに従う。
- 恐怖、暴力、差別、性的内容、危険行為、個人情報を含めない。
- 特定の家庭構成、経済環境、地域文化、専門知識を正答の前提にしない。
- 参照教材の人物、固有の出来事、文章表現を再現しない。
- 目標、小さく観察可能な失敗、主人公が選ぶ確認行動、やり直した結果を因果でつなぐ。
- 性格語、失敗を見られた場面、直後の反応、心情推論の根拠を別々の文へ配置できる設計にする。
- 実在教材を転載・言い換えしない独自の合成例を構造見本として使用する。

### 可変入力

- 対象学年
- 題材カテゴリ
- 本文長
- 生成条件プロファイル
- seed相当の候補識別子
- 使用可能な舞台・人物属性・感情・行動の制約
- 禁止題材と直近候補の署名

### 教材コンテキスト

`story-plan-context.v1` では、次を可変入力と一緒にサーバー側から渡す。

- 対象学年までの文科省配当漢字候補（1年80字、2年累積240字、3年累積440字）とDB版・候補状態
- 現行の `prototype_lexicon` の表記・読み・暫定語彙学年
- 教育基本語彙の低学年・高学年候補DBは人間未確認で、限定 `prototype_lexicon` の監査投影にだけ接続され、生成語の選択や学年適合を保証できないという制限
- `ANCHOR-RUBY-Q12` と `ANCHOR-RUBY-Q18` から抽出した「語彙難度と表記難度を分ける」「初出だけ読みを示す」という抽象的特徴

配当漢字一覧は下流の表記検証用の参考であり、かな限定の物語設計図へ漢字を出力させる目的ではない。早稲アカの本文、人物、固有の出来事、選択肢をAPIへ例文として送らない。監査投影は限定語彙が低学年候補母集団に一致することだけを示す。出典・権利・版・対象学年を人間確認した生成用語彙は、別の昇格ゲートを通して同じコンテキスト入口へ追加する。

プロンプトはコード内で版管理し、`prompt_version` と内容ハッシュを記録する。モデル設定画面上だけで変更する運用にしない。

## 7. 検証ゲート

モデル候補は次の順で検査する。一つでも不合格なら配布データへ含めない。

1. JSON Schemaに一致する。
2. 拒否、途中終了、APIエラーではない。
3. 必須の物語構造が成立している。
4. カテゴリ、舞台、人物、出来事の組合せが矛盾しない。
5. 年齢不相応、危険、恐怖、偏見、個人情報を含まない。
6. 正式DBまたはプロトタイプ限定語彙へ写像できる。
7. 標準4問の正答根拠を配置できる。
8. 抜き出し正答と選択式正答を一意にできる。
9. 既存候補や参照教材との表層・物語骨格の重複が閾値未満である。
10. 既存コーパステストに組み込んでも全条件を満たす。

自動検査に合格した候補の状態は `automated_checks_passed` までとする。人間レビューなしで `reviewed_practice` または `release_ready` へ進めない。

## 8. 再現性と保存項目

文章生成モデルの出力は本質的に変動し、同じモデル系列でもスナップショット変更によって挙動が変わり得る。seedだけで完全再生成できるとは扱わない。

ライブAI生成でseedは必須の決定乱数ではないが、候補ID、入力条件のハッシュ、キャッシュ、来歴、変化のヒントを結び付けるため保持する。AI生成を選択中のランダムseedボタンは値を変えるだけとし、モデル呼出しは明示した生成ボタンでのみ行う。完全再現には、検証済みstory plan自体と、モデル・プロンプト・コンテキスト・スキーマ・validatorの各版を保存する。

採用候補には次を保存する。

```text
provider
API endpoint
model IDまたは固定スナップショットID
reasoning・生成設定
prompt_versionとprompt_hash
schema_version
入力条件と候補識別子
OpenAI request ID
取得日時
生レスポンスのSHA-256
正規化済みstory plan
validator_version
検査結果
人間レビュー状態
```

完全に同じ問題を再表示するときはAPIを再呼出しせず、保存済みstory planと既存generator版から固定item revisionを読み込む。新しい同等問題を作る場合だけ、新しい候補識別子でモデルを呼び出す。

## 8.1 AI出力の再利用とテンプレート化

AIで生成した候補のうち、検証・レビューで良問と判断されたものは、次回以降のアルゴリズム生成へ再利用できる。ただし、完成本文をそのまま複製するのではなく、本文から検証済みの物語構造と出題構造を抽出し、パラメータ化したテンプレートとして保存する。

```text
raw model response              改変せず保存。再配布・直接表示しない
        ↓ normalize / validate
story-plan candidate            機械検査済みの設計図。まだ採用しない
        ↓ self review / calibration
approved story template         検証済み構造。seedで多様な問題へ展開
        ↓ deterministic generator
item revision                   本文・設問・正答・根拠を固定した問題
```

### テンプレートに保存するもの

- `template_id`、`template_version`、`template_status`
- 対応する `blueprint_id`、`story_structure_id`、設計図スキーマ版、学年範囲、カテゴリ、本文長、難度プロファイル
- 登場人物、舞台、目的、失敗、心情変化、解決の抽象スロット
- 正答根拠の配置、問ごとの測定能力、許容する根拠距離
- 使用可能・禁止の語彙カテゴリ、人物属性、出来事の組合せ制約
- 生成元の `candidate_id`、採用した `item_id`、レビュー・校正記録への参照
- `template_hash`、抽出器・正規化器・検証器の版

テンプレートには個人情報、未検証の自由記述、APIキー、モデルの生レスポンスを含めない。参照教材やAI出力の特徴的な文章表現・固有名詞・展開をそのまま再利用せず、抽象化できない候補はテンプレートへ昇格させない。

### テンプレートの状態

```text
candidate → automated_template_passed → reviewed_template
          → calibrated_template → approved_template → retired_template
```

`approved_template` だけを通常のアルゴリズム生成で利用する。自動検査だけでテンプレートを承認しない。元となるitemが `needs_revision`、`rejected`、`withdrawn` になった場合、その理由が構造に影響するかを確認し、影響するテンプレートを利用停止する。

### 再利用時の決定性と多様性

`approved_template`、固定DB版、アルゴリズム版、生成条件、seedから本文・設問・正答・根拠を決定的に生成する。再利用時にモデルを再呼出ししないため、API停止後も同一問題を再表示できる。

同一テンプレートの過剰反復を避けるため、生成済みitemの署名（登場人物、舞台、出来事、表現の特徴）を保存し、類似度閾値、カテゴリ別使用回数、直近利用期間を選択条件へ含める。テンプレート由来の問題には、`template_id` と `template_revision` を必ず生成来歴へ残す。

### AI候補をテンプレートへ昇格する条件

1. `story-plan.v1` と `story-standard-4q.v1`／`story-retry-craft.v1` の組合せ契約、および既存の自動検査を通過する。
2. 既存の標準4問で正答根拠と一意性が成立する。
3. 開発者レビュー、答えを隠した解き直し、表現・公平性確認を通過する。
4. 児童試行を行う場合は、テンプレート単位の問題群として偏りと再現性を確認する。
5. 採用理由、却下理由、派生itemの評価結果を台帳へ記録する。

この構成により、AIは候補を増やす役割を担い、安定した品質と再現性はテンプレート・決定的ジェネレータ・レビュー台帳が担う。

## 9. APIキーと秘密情報

- `OPENAI_API_KEY` は開発端末、CI、サーバーレス環境の秘密変数に保存する。
- HTML、ブラウザJavaScript、静的JSON、Git管理対象、ログへ含めない。
- ブラウザのlocalStorage、IndexedDB、URLパラメータへ保存しない。
- APIキーを利用者へ入力させる画面を作らない。
- 開発用と公開用でOpenAI Projectと利用上限を分離する。
- 誤って露出したキーは直ちに無効化し、履歴から除去する。

## 10. 未成年者向けのデータ方針

[OpenAIのUnder 18 API Guidance](https://developers.openai.com/api/docs/guides/safety-checks/under-18-api-guidance) に従い、未成年者向けには通常より強い安全性・プライバシー措置を取る。

初期導入では、APIへ送る情報を開発者が作成した一般的な生成条件だけに限定する。

### APIへ送らないもの

- 子どもの氏名、学校名、住所、連絡先
- 子どもの自由記述回答
- 正誤、点数、所要時間、行動履歴
- 保護者メモや個別の学習相談
- 端末・アカウントを児童へ結び付ける識別子

13歳未満または地域のデジタル同意年齢未満の児童の個人データを扱う設計へ変更する場合は、法令確認とOpenAI APIのZero Data Retention要件を含む別のプライバシー審査を完了するまで実装しない。

## 11. 障害時と不合格時の扱い

API障害、レート制限、拒否、スキーマ不一致、内容検査不合格が起きても、未検証の出力を画面へ表示しない。

```text
APIまたは候補検査が失敗
        ↓
限定回数だけ再試行
        ↓ 失敗
保存済みstory plan packへフォールバック
        ↓
既存の決定的ジェネレータで問題を生成
```

ライブ生成の利用可否を、教材を利用できるかどうかの単一障害点にしない。

## 12. 段階的な実装手順

### 段階A：スキーマと評価の試作

1. `story-plan.v1` の共有JSON Schemaと追加検証器を作る。
2. 固定プロンプトと20件程度の入力fixtureを作る。
3. 通常のResponses APIで少数候補を生成する。
4. 既存ジェネレータへ接続せず、候補JSONだけを検査する。
5. 不自然さ、重複、年齢不適合、物語構造不足を記録する。

### 段階B：事前生成パイプライン

1. 候補生成、取込、正規化、検証、配布JSON生成を別スクリプトにする。
2. 既存コーパステストへstory plan packを追加する。
3. APIを呼ばなくても全テストとブラウザ表示が動くようにする。
4. 数百件以上へ増やす段階でBatch APIを評価する。

### 段階C：人間レビュー

1. カテゴリごとの多様性と重複率を確認する。
2. モデル生成であることを伏せず、却下理由を記録する。
3. 合格した候補だけを固定版へ昇格する。
4. 現行テンプレート版と比較し、自然さと正答根拠の維持を評価する。

### 段階D：ライブ生成の開発評価

ローカルプロキシは候補収集と比較のために実装済みである。公開ライブ生成へ進むかは、事前生成での品質、予算上限、監視、利用規約、未成年者向け安全措置を評価してから別途判断する。

## 13. 初期導入の完了条件

- APIキーがブラウザとGit管理対象へ存在しない。
- モデル出力が`story-plan.v1`へ適合する。
- すべての候補が既存検証を通過し、不合格候補が配布物へ混入しない。
- API停止中でも既存の問題生成とテストが動く。
- 採用候補をモデル・プロンプト・スキーマ・検査版まで追跡できる。
- 保存済みitem revisionを完全に再表示できる。
- 児童の個人情報をAPIへ送らない。
- 人間レビュー前の候補が児童向け・評価用として表示されない。
